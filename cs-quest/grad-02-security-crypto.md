## 📖 講義 1: 要塞の設計思想 — 脅威モデルと原則(20 XP)

セキュリティは「絶対安全」を作る技術ではない。**「誰から、何を、どこまで守るか」(脅威モデル)** を定め、
攻撃コストを割に合わなくする工学だ。

要塞の鉄則:
- **多層防御 (defense in depth)**: 1 枚の壁に頼らない
- **最小権限 (least privilege)**: 各部品に必要最小限の権力だけ
- **フェイルセキュア**: 壊れたら「閉じる」方向に倒れる
- **ケルクホフスの原理**: **アルゴリズムは公開してよい。秘密は鍵だけにせよ**(隠蔽による安全は脆い)
- 「自前の暗号を実装するな」: 枯れた標準ライブラリを使う。手製の暗号は必ず破られる

セキュリティが守る 3 本柱 **CIA**: 機密性 (Confidentiality)、完全性 (Integrity)、可用性 (Availability)。

## 📖 講義 2: 同じ鍵の守り — 対称鍵暗号(20 XP)

送受信者が**同じ秘密鍵**で暗号化・復号(World 9 の合同算術の世界)。

- **AES**: 現代の標準ブロック暗号。128 ビットブロックを鍵で何重にも撹拌
- **動作モード**が死活的に重要: **ECB は使うな**(同じ平文ブロックが同じ暗号文になり、絵柄が透ける有名な失敗)。
  CBC や CTR、認証付きの **GCM** を使う
- 速い。だが**「鍵をどうやって相手に渡すか」**という根源問題が残る → 講義 3 へ

## 📖 講義 3: 二つの鍵の魔法 — 公開鍵暗号と RSA(20 XP)

**公開鍵で施錠、秘密鍵で開錠**。公開鍵は世界に配ってよい。鍵配送問題が解ける。

### RSA を数論から組む(World 9 の道具だけで作れる!)

1. 大きな素数 p, q を選び n = pq、φ(n) = (p−1)(q−1)
2. e を選び(公開)、d ≡ e⁻¹ (mod φ(n)) を**拡張ユークリッド**で計算(秘密)
3. 暗号化 c = mᵉ mod n、復号 m = cᵈ mod n(**繰り返し二乗法**で高速に)

**安全性の根拠**: n から p, q を**素因数分解するのが難しい**こと(効率的な古典アルゴリズムが未発見)。
World 13 の言葉では「素因数分解は P にあると分かっていない」。
⚠️ だが**量子計算機のショアのアルゴリズムは多項式時間で素因数分解する** → 耐量子暗号(格子暗号)へ移行が進行中。

実用では RSA は遅いので「公開鍵で**共通鍵を交換**し、本文は AES」というハイブリッド(World 8 の HTTPS の正体)。

## 📖 講義 4: 指紋と封蝋 — ハッシュ・署名・認証(20 XP)

- **暗号学的ハッシュ(SHA-256 等)**: 一方向(原像計算困難)、衝突発見困難。
  ⚠️ World 9 の誕生日攻撃により、衝突発見は 2^(n/2) で起こりうる → だから出力は 256 ビット必要
- **パスワード保存**: 暗号化ではなく**ソルト付きハッシュ**(bcrypt/argon2 のような**わざと遅い**関数)。
  ソルトはレインボーテーブルを無効化し、低速性は総当たりを高コストにする
- **MAC / HMAC**: 共通鍵で「改ざんされていない」ことを保証(完全性 + 認証)
- **デジタル署名**: 秘密鍵で署名、公開鍵で検証。「本人が書いた・改ざんされていない」を公開検証可能に。
  だが「その公開鍵が本当に本人のものか?」→ **証明書(CA)と信頼の連鎖**(World 8 の HTTPS 証明書)

## 📖 講義 5: 鍵を交わす儀式 と システムの綻び(20 XP)

### ディフィー・ヘルマン鍵交換
盗聴者の目の前で、共有秘密を作る魔法。離散対数の困難性に基づく。
**前方秘匿性 (forward secrecy)**: セッションごとに使い捨て鍵を作れば、後で長期鍵が漏れても過去の通信は守られる。

### システム攻撃の代表(守るために知る)

| 攻撃 | 仕組み | 防御 |
|---|---|---|
| **SQL インジェクション** | 入力を SQL 文に連結し構文を乗っ取る | **プリペアドステートメント**(データとコードを分離) |
| **XSS** | 悪意あるスクリプトを他人のブラウザで実行 | 出力エスケープ、CSP |
| **バッファオーバーフロー** | 配列の境界を超えて書き、リターンアドレスを乗っ取る(World 7/12 のスタックフレーム) | 境界チェック、ASLR、スタックカナリア、メモリ安全な言語(Rust) |
| **CSRF** | 被害者のログイン状態を悪用した偽リクエスト | トークン、SameSite Cookie |

**共通の教訓**: ほとんどの攻撃は「**データとコードの境界が曖昧**」なところに生まれる。
入力は信用しない(World 13 が示すとおり「完全な入力検証器」は作れないので、構造的に分離する)。

## ⚔️ クエスト

### クエスト 1: 騎士の心得(40 XP)

1. 「我が社の暗号は独自方式なので解読不可能」という売り文句のどこが危険か(ケルクホフス)。
2. なぜパスワードは「ハッシュ化」して保存し、「暗号化」して保存しないのか。ソルトの役割は?
3. ECB モードで画像を暗号化すると元の絵柄が透ける理由を述べよ。

<details>
<summary>📜 答え</summary>

1. 安全性をアルゴリズムの秘匿に依存させている。アルゴリズムは必ず漏れる(逆コンパイル・退職者・解析)もので、漏れた瞬間に全崩壊する。安全性は鍵だけに依存させ、アルゴリズムは公開審査に耐えたものを使うべき。
2. 暗号化は復号鍵があれば元に戻せるため、鍵が漏れれば全パスワードが露出する。ハッシュは一方向で、サーバですら元のパスワードを知る必要がない。ソルトは各ユーザーに異なる値を加え、同じパスワードでも異なるハッシュにして、事前計算(レインボーテーブル)と一括攻撃を無効化する。
3. ECB は同じ平文ブロックを常に同じ暗号文ブロックに写すため、画像の同色領域がそのままパターンとして暗号文に残る。
</details>

### クエスト 2: 💻 実装クエスト「教科書 RSA を組む」(80 XP)

World 9 の拡張ユークリッドと繰り返し二乗法を使い、**小さな素数**で RSA を一から実装せよ
(鍵生成 → 暗号化 → 復号 → 元に戻ることを確認)。
完成したら**コメントに必ず書け**:「これは学習用。実運用では教科書 RSA は安全でない
(パディング OAEP・乱数・サイドチャネル対策が必要)。本番は標準ライブラリを使う」。

### クエスト 3: 💻 攻防クエスト「注入を塞ぐ」(60 XP)

文字列連結で SQL を組む脆弱なコード(`"SELECT * FROM users WHERE name='" + input + "'"`)を用意し、
`' OR '1'='1` で全件が漏れることを**自分のローカル DB(World 11 の SQLite)**で再現せよ。
その後プリペアドステートメントに直し、同じ入力が無害化されることを確認せよ。
(⚠️ 自分の環境のみ。他者のシステムへの試行は不可)

### クエスト 4: プロトコルの目(40 XP)

1. ディフィー・ヘルマンで、盗聴者が通信を全部見ても共有鍵を計算できないのはなぜか(困難性の名前を挙げよ)。
2. 前方秘匿性があると、サーバの長期秘密鍵が将来漏れても過去の通信が守られるのはなぜか。
3. デジタル署名と MAC の違いを 1 点述べよ(誰が検証できるか)。

<details>
<summary>📜 答え</summary>

1. やり取りされる g, gᵃ, gᵇ から共有鍵 g^(ab) を求めるには離散対数を解く必要があり、効率的な古典アルゴリズムが知られていない(離散対数問題の困難性)。
2. セッションごとに使い捨ての一時鍵で実際の暗号化鍵を作り、セッション後に破棄するため。長期鍵は一時鍵の認証にしか使われず、過去のセッション鍵を復元する材料にならない。
3. MAC は共通鍵を持つ者だけが検証でき、検証者も偽造可能(送受信者が対等)。署名は秘密鍵を持つ署名者だけが作れ、公開鍵を持つ**誰でも**検証できる(否認防止になる)。
</details>

## 👹 ボス戦: 要塞の城主「サイファー・ガーディアン」(200 XP)

ノートを閉じて挑め。7 割以上で勝利。

1. ケルクホフスの原理と、それに反する「隠蔽によるセキュリティ」の問題を述べよ。
2. 対称鍵暗号の鍵配送問題を、公開鍵暗号がどう解決するか説明せよ。
3. RSA の安全性が依存する数学的困難性と、それを脅かす将来の技術を挙げよ。
4. 暗号学的ハッシュに必要な 3 性質を述べ、誕生日攻撃が出力長に与える要請を説明せよ。
5. パスワード保存の正しい方法を「ハッシュ・ソルト・低速関数」の 3 語を使って説明せよ。
6. SQL インジェクションとバッファオーバーフローに共通する根本原因を 1 文で述べ、それぞれの防御を挙げよ。
7. HTTPS 接続確立で「暗号化・認証・鍵交換・完全性」がどう組み合わさるか、World 8 の知識と統合して説明せよ。

<details>
<summary>📜 解答の骨子</summary>

1. 安全性は鍵のみに依存させるべきという原則。アルゴリズムの秘匿に頼ると、必ず起きる漏洩で全面崩壊し、公開審査による弱点発見の機会も失う。
2. 公開鍵で共通鍵(セッション鍵)を暗号化して送るか、DH で鍵を共有することで、秘密を事前共有せずに対称鍵を安全に確立する。
3. 大きな合成数の素因数分解の困難性。量子計算機のショアのアルゴリズムが多項式時間で解くため、耐量子(格子ベース)暗号への移行が必要。
4. 原像計算困難・第二原像計算困難・衝突発見困難。誕生日攻撃で衝突は約 2^(n/2) で見つかるため、128 ビット安全性には 256 ビット出力が要る。
5. パスワードにユーザー固有のソルトを連結し、bcrypt/argon2 などの意図的に低速なハッシュ関数を多数回適用して保存する。ソルトで一括・事前計算攻撃を防ぎ、低速性で総当たりコストを上げる。
6. 共通原因: データとコード(命令)の境界が曖昧で、入力が制御フローや構文として解釈されること。防御: SQL はプリペアドステートメント(データ/コード分離)、メモリ破壊は境界チェック・ASLR・カナリア・メモリ安全言語。
7. DNS で IP 解決 → TCP 接続 → TLS ハンドシェイクで証明書により**サーバを認証**、(EC)DHE で**鍵交換**(前方秘匿性)、確立した共通鍵で AES-GCM により本文を**暗号化+完全性保証**。以降の HTTP はこの暗号路上を流れる。
</details>
