## 📖 講義 1: 魔法詠唱の正体 — プログラムはどう実行されるか(10 XP)

君の書いたコードが動くまでの旅:

```
ソースコード → [コンパイラ] → 機械語 → [CPU] フェッチ → デコード → 実行 → 書き戻し
```

- **コンパイラ**: 事前に機械語へ一括翻訳(C, C++, Rust, Go)。実行が速い
- **インタプリタ**: 実行時に逐次解釈(素の Python)。手軽だが遅い
- **中間方式 / JIT**: バイトコードに変換し、仮想マシンが実行しつつ熱い部分を機械語化(Java, JavaScript V8)

**CPU の一生は 4 拍子の繰り返し**: 命令を**取って(fetch)**、**読み解き(decode)**、**実行し(execute)**、結果を**書き戻す(write back)**。
現代の CPU はこれをパイプライン化・並列化して 1 秒に数十億回繰り返す。

## 📖 講義 2: 都市の倉庫網 — メモリ階層とキャッシュ(10 XP)

CPU から見ると、データ置き場は「近くて狭い」か「遠くて広い」かのトレードオフでできている。

| 倉庫 | 容量の目安 | 速度の目安(体感換算※) |
|---|---|---|
| レジスタ | 数百バイト | 手元の机(1 秒) |
| L1 キャッシュ | 〜64 KB | 同じ部屋の棚(数秒) |
| L2 / L3 キャッシュ | 〜数十 MB | 隣の建物(数十秒〜数分) |
| メインメモリ (RAM) | 数十 GB | 隣町まで買い出し(数時間) |
| SSD / ディスク | 数 TB | 海外取り寄せ(数週間〜数年) |

※ 1 CPU サイクルを 1 秒に引き伸ばした場合のたとえ。

**キャッシュの賭け** — 局所性 (locality):
- **時間的局所性**: さっき使ったデータはまたすぐ使うだろう
- **空間的局所性**: 使ったデータの**隣**もすぐ使うだろう → メモリは**キャッシュライン(64 バイト程度)単位**でまとめて運ばれる

**だから**: 配列を先頭から舐めるのは爆速(隣が次々ヒット)、連結リストを辿るのは遅い(ポインタの先は遠くの倉庫)。
World 4 で「クイックソートは実測最速級」と言った理由がこれだ —— 連続メモリを順に触るからキャッシュに優しい。

## 📖 講義 3: 都市の働き手たち — プロセスとスレッド(10 XP)

| | プロセス | スレッド |
|---|---|---|
| 正体 | 実行中のプログラム | プロセス内の実行の流れ |
| メモリ空間 | **独立**(他のプロセスから見えない) | **同じプロセス内で共有** |
| 持ち物 | アドレス空間、ファイルハンドル、ヒープ | 自分のスタックとレジスタ(+プログラムカウンタ) |
| 生成・切替コスト | 重い | 軽い |
| 1 つが死ぬと | 他のプロセスは無事 | プロセスごと巻き込まれうる |

- **コンテキストスイッチ**: OS がCPU を別の実行主体に切り替えること。レジスタ等の状態を保存・復元する
- **並行 (concurrency)**: 複数の仕事を切り替えながら進める / **並列 (parallelism)**: 物理的に同時に実行する
- **共有メモリの罠 — データ競合**: 2 つのスレッドが同じ変数に同時に書くと結果が壊れる。
  **ロック(mutex)** で守るが、今度は互いに相手のロックを待ち合う**デッドロック**の危険が生まれる
- デッドロックの 4 条件(相互排他・保持と待機・横取り不可・循環待機)— 1 つでも崩せば防げる

📺 [プロセスとスレッドの違い(動画)](https://www.youtube.com/watch?v=4rLW7zg21gI)

## 📖 講義 4: 都市の奇妙な数体系(10 XP)

### 浮動小数点数 — 「0.1 + 0.2 ≠ 0.3」の謎

IEEE 754 では実数を `符号 × 仮数 × 2^指数` の 2 進数で表す。
**0.1 は 2 進数では無限循環小数**になるため、正確に格納できない。だから:

```python
>>> 0.1 + 0.2
0.30000000000000004
```

**冒険者の掟**: 浮動小数点を `==` で比較するな。`|a − b| < ε` で比較せよ。金額は整数(セント単位)か十進小数型で持て。

### Unicode と UTF-8 — 世界の文字を運ぶ仕組み

- **Unicode**: 世界中の文字に番号(コードポイント)を割り当てた表。「あ」は U+3042
- **UTF-8**: コードポイントをバイト列にする**可変長**エンコーディング。ASCII は 1 バイト、日本語はおおむね 3 バイト
- **掟**: 「文字数 ≠ バイト数」。入出力の境界では必ずエンコーディングを意識せよ

### エンディアン — バイトの並べ方の流儀

32 ビット整数 `0x12345678` をメモリに置くとき:
- **ビッグエンディアン**: `12 34 56 78`(上位から)— ネットワークバイトオーダー
- **リトルエンディアン**: `78 56 34 12`(下位から)— x86 などの主流 CPU

異なる流儀のマシン間で生のバイト列をやり取りすると数値が化ける。ネットワーク越しでは変換(htonl 等)が要る。

## ⚔️ クエスト

### クエスト 1: 倉庫番の目利き(20 XP)

次の 2 つのコードはどちらも n×n 行列の総和を計算する。どちらが速いか、理由とともに答えよ。

```c
// (a) 行優先                          // (b) 列優先
for (i = 0; i < n; i++)               for (j = 0; j < n; j++)
  for (j = 0; j < n; j++)               for (i = 0; i < n; i++)
    sum += A[i][j];                        sum += A[i][j];
```

<details>
<summary>📜 答え</summary>

(a) が速い(C の 2 次元配列は行優先で連続配置)。(a) は連続アドレスを順に読むのでキャッシュラインが有効活用される。(b) は毎回 n 要素先へ飛ぶためキャッシュミスが多発する。**同じ O(n²) でも実測は数倍違う。**
</details>

### クエスト 2: 働き手の問答(20 XP)

1. ブラウザのタブが 1 つ固まっても他のタブが生きているのはなぜか(プロセス/スレッドの言葉で)。
2. 2 スレッドが `counter += 1` を 1000 回ずつ実行したのに、結果が 2000 にならないことがある。何が起きているか。
3. デッドロックの 4 条件のうち「循環待機」を崩す実用的な方法を 1 つ挙げよ。

<details>
<summary>📜 答え</summary>

1. 現代のブラウザはタブごとに**プロセス**を分けている。プロセスはメモリ空間が独立しているため、1 つの暴走が他に波及しない。
2. **データ競合**。`counter += 1` は読み出し→加算→書き戻しの 3 ステップで、2 スレッドの読み出しが交錯すると片方の加算が失われる(lost update)。ロックかアトミック命令が必要。
3. **ロックの獲得順序を全スレッドで統一する**(常に A → B の順で取る)。循環が構造的に起きなくなる。
</details>

### クエスト 3: 奇妙な数の観測(20 XP)

手元の言語で次を確かめ、それぞれ理由を説明せよ。

1. `0.1 + 0.2 == 0.3` の結果
2. `0.5 + 0.25 == 0.75` の結果(なぜこちらは真になる?)
3. 文字列 `"こんにちは"` の文字数と UTF-8 でのバイト数

<details>
<summary>📜 答え</summary>

1. 偽。0.1 も 0.2 も 2 進では無限循環小数で丸め誤差を含む。
2. 真。0.5 = 2⁻¹、0.25 = 2⁻² は 2 進で正確に表せるため誤差がない。
3. 5 文字、15 バイト(ひらがなは UTF-8 で 3 バイト)。
</details>

### クエスト 4: 💻 実装クエスト「ビットの鍛錬」(50 XP)

ビット演算で次を実装せよ(World 7 の住人はみな 2 進数で話す):

1. 整数 n の i ビット目を 取得 / セット / クリア / 反転する 4 関数
2. 整数の 2 進表現に含まれる 1 の個数を数える(popcount)
3. n が 2 のべき乗かを 1 行で判定する

<details>
<summary>📜 ヒント</summary>

3 は `n > 0 && (n & (n - 1)) == 0`。2 のべき乗は 1 が 1 個しか立っていないことを利用。
</details>

## 👹 ボス戦: 地下都市の心臓「マシンゴーレム」(100 XP)

ノートを閉じて挑め。7 割以上で勝利。

1. コンパイラとインタプリタの違いを、実行までの流れの図とともに説明せよ。
2. メモリ階層を速い順に 4 段以上挙げ、キャッシュが効く理由を「局所性」の 2 種類で説明せよ。
3. プロセスとスレッドの違いを 3 点挙げよ。
4. 配列の線形走査が連結リストの走査より実測で大幅に速い理由は?(計算量は同じ O(n) なのに)
5. 浮動小数点の比較で `==` を避けるべき理由と代替手段は?
6. リトルエンディアンのマシンで `0xAABBCCDD` をメモリダンプするとどう並ぶ?

<details>
<summary>📜 解答</summary>

1. コンパイラは実行前に全体を機械語へ翻訳して保存、実行時は機械語を直接走らせる。インタプリタは実行時にソース(またはバイトコード)を逐次解釈する。
2. レジスタ → L1 → L2/L3 → RAM → SSD/ディスク。時間的局所性(同じデータの再利用)と空間的局所性(隣接データの利用)が高いため、小さく速い記憶に最近の周辺データを置くだけでヒット率が高くなる。
3. メモリ空間(独立 / 共有)、生成・切替コスト(重い / 軽い)、障害の波及(隔離される / プロセス全体に及ぶ)。
4. 配列は連続メモリでキャッシュライン単位の先読みが効くが、連結リストはノードが散らばっておりほぼ毎回キャッシュミス+ポインタ追跡になるから。
5. 2 進で表せない値の丸め誤差により、数学的に等しいはずの計算結果がビット単位では一致しないため。`|a − b| < ε`(または ULP 比較)を使う。
6. `DD CC BB AA`(下位バイトが先頭アドレスに置かれる)。
</details>
