> **実際のサービスを丸ごと設計する**場所だ。

## 📖 システム設計の「型」(全問題で使う手順)

実際の設計面接でもこの順で進めると強い。**いきなり図を描かない。**

1. **要件を確認する(機能要件・非機能要件)**
   - 機能: 何ができるべきか(投稿する、検索する…)
   - 非機能: 規模・遅延・可用性・一貫性の要求は?
2. **規模を概算する(封筒の裏計算 / back-of-the-envelope)**
   - 1 日のアクティブユーザー、QPS(秒あたりリクエスト)、ストレージ、帯域
3. **API を設計する**(エンドポイントと引数)
4. **データモデルを設計する**(テーブル/スキーマ、どの DB を選ぶか)
5. **全体構成を描く**(クライアント → LB → アプリ → キャッシュ → DB)
6. **ボトルネックを潰す**(スケール、キャッシュ、シャーディング、レプリケーション)
7. **トレードオフを明言する**(CAP のどちらを取ったか、なぜ)

### 概算に使う暗記数字(これだけ覚える)

| 量 | 目安 |
|---|---|
| 1 日の秒数 | 約 86,400 ≈ 10 万 |
| メモリ参照 | 約 100 ナノ秒 |
| SSD ランダム読み | 約 100 マイクロ秒 |
| ディスクシーク | 約 10 ミリ秒 |
| 同一データセンター内往復 | 約 0.5 ミリ秒 |
| 大陸間往復 | 約 100 ミリ秒 |
| 1 文字(UTF-8 英数) | 1 バイト |

**QPS の概算式**: `1 日のリクエスト数 ÷ 86,400`。ピークは平均の 2〜3 倍を見込む。

## 問題 1: URL 短縮サービス(TinyURL)🟢

「長い URL を短い URL にして、アクセスすると元に転送する」サービスを設計せよ。

### ステップ 1: 要件

- **機能**: ①長い URL → 短い URL を生成 ②短い URL → 元 URL へリダイレクト ③(任意)有効期限、カスタム別名、統計
- **非機能**: 読み込み(リダイレクト)が圧倒的多数。**読み >> 書き**。低遅延必須。短縮URLは衝突してはならない

### ステップ 2: 概算

- 書き込み 1 億 URL/月 ≈ 40 URL/秒、読み込みはその 100 倍として 4,000 QPS
- 1 レコード ≈ 500 バイト × 1 億 × 12 か月 × 5 年 ≈ 3 TB。十分 DB に収まる規模
- **示唆**: 読み多寄りなのでキャッシュが効く。短いキーで一意性を保つのが核心

### ステップ 3: API

```
POST /shorten        body: { long_url }            -> { short_url }
GET  /{short_code}                                  -> 301 リダイレクト
```

### ステップ 4: データモデル

```
url_mapping
  short_code  VARCHAR(7)  PRIMARY KEY   -- 例: "aZ3xK9p"
  long_url    TEXT
  created_at  TIMESTAMP
  expires_at  TIMESTAMP NULL
```

**短いコードの作り方(設計の核心)**:
- **方式A: ハッシュ + 切り詰め** — long_url を MD5 し先頭を Base62(a–z A–Z 0–9 の 62 文字)で 7 文字に。衝突したら付け直す。長所: シンプル。短所: 衝突処理が要る
- **方式B: 連番 + Base62 変換** — グローバルなカウンタ(連番)を Base62 にエンコード。`62^7 ≈ 3.5 兆`通りで枯渇しない。衝突なし。短所: 連番が予測可能(セキュリティ上は乱数化 or 範囲を分散ID生成器で配る)
- **採用**: 方式B(衝突処理が不要で確実)。連番の発番は採番サービス(後述のID生成)に任せる

### ステップ 5: 全体構成

```
クライアント → ロードバランサ → アプリサーバ → キャッシュ(Redis) → DB
                                              ↘ なければ DB を引いてキャッシュに載せる
```

リダイレクトは「short_code でキャッシュを引く → あれば即返す → なければ DB → キャッシュに載せる」。読みの 9 割以上はキャッシュで完結する。

### ステップ 6: ボトルネックと対策

- **読みの集中**: Redis キャッシュ + CDN。人気 URL はキャッシュヒット率が高い(World 7 の局所性)
- **DB のスケール**: short_code でシャーディング(範囲 or ハッシュ)。3 TB なら当面単一でも可
- **採番の単一障害点**: ID 生成器を冗長化(後述の Snowflake 方式)

### ステップ 7: トレードオフ

- リダイレクトは **301(恒久)** だとブラウザがキャッシュして統計が取れない。統計が欲しいなら **302(一時)**。可用性重視で **AP 寄り**(短縮URLは多少古くても致命的でない)

**面接での一言**: 「読み多寄りなのでキャッシュが主役です。一意キーは連番の Base62 変換で衝突を構造的に排除します。」

## 問題 2: レートリミッター(APIの流量制限)🟡

「ユーザーごとに 1 分間に 100 リクエストまで」を強制する仕組みを設計せよ。

### 要件と概算

- ある識別子(ユーザーID/IP/APIキー)ごとに、単位時間あたりの回数を制限
- **超高速**(全 API の手前を通るので遅延が命)、**分散環境で一貫**して効くこと

### アルゴリズムの選択肢(これが設計の核)

| 方式 | 仕組み | 特徴 |
|---|---|---|
| 固定窓カウンタ | 「1 分ごと」のカウンタを増やす | 単純だが窓の境界で 2 倍流れる問題 |
| スライディングログ | 各リクエスト時刻を記録し、窓内の数を数える | 正確だがメモリを食う |
| **スライディング窓カウンタ** | 現窓と前窓のカウントを加重平均 | 精度とコストのバランスが良い |
| **トークンバケツ** | 一定レートでトークンを補充、1 リクエスト=1 トークン消費 | バースト(瞬間的な集中)を許容。実務で最も人気 |

**採用: トークンバケツ**。「バケツ容量 = バースト許容量」「補充レート = 平常の制限」を別々に設定でき、現実のニーズに合う。

```python
# 概念実装(実際は Redis 上でアトミックに行う)
import time

class TokenBucket:
    def __init__(self, capacity, refill_per_sec):
        self.capacity = capacity            # バケツの最大トークン数
        self.refill = refill_per_sec        # 1 秒あたり補充数
        self.tokens = capacity
        self.last = time.time()

    def allow(self):
        now = time.time()
        # 経過時間ぶんトークンを補充(上限まで)
        self.tokens = min(self.capacity, self.tokens + (now - self.last) * self.refill)
        self.last = now
        if self.tokens >= 1:
            self.tokens -= 1
            return True                     # 通す
        return False                        # 制限(429 を返す)
```

### 分散環境での実装

- 状態(残トークン)は **Redis** に置き、全アプリサーバが共有する
- 競合を防ぐため **Lua スクリプト**で「読み→計算→書き」をアトミックに実行(World 11 の並行制御、World 10 の臨界区間の実戦)
- 制限超過時は **HTTP 429 Too Many Requests** を返す

### トレードオフ

- Redis 往復の遅延 vs 各サーバローカルでの近似(ローカルだと厳密性が落ちるが速い)。厳密さを取るなら集中管理、速度を取るならローカル+定期同期

## 問題 3: ニュースフィード(Twitter/Instagram 風)🔴

「フォローした人の投稿が時系列で並ぶフィード」を設計せよ。最難関の定番。

### 要件と概算

- **機能**: 投稿する、フォローする、自分のフィードを見る
- **規模**: 2 億 DAU、1 人平均 200 フォロー、読み:書き = 100:1 の**読み多寄り**
- **核心の難所**: フィード生成は「フォロー中**全員**の投稿をマージして時系列ソート」。これを毎回やると重い

### 設計の二大流派(ここが勝負どころ)

**① プル型(読み時に生成 / fan-out on read)**
- フィードを見るたびに、フォロー中全員の最新投稿を取得しマージ
- 長所: 書き込みが軽い、ストレージ節約。短所: **読みが重い**(フォロー数が多いと遅い)

**② プッシュ型(書き込み時に配布 / fan-out on write)**
- 誰かが投稿したら、その**全フォロワーのフィード(キャッシュ)に即座に書き込む**
- 長所: 読みが爆速(自分のフィードを読むだけ)。短所: **書きが重い**、有名人問題

**③ ハイブリッド(実際の Twitter 等の解)**
- 一般ユーザー: プッシュ型(フォロワーが少ないので配布が軽い)
- **有名人(数百万フォロワー): プル型**(全員に配ると書き込みが爆発するため、読み時にマージ)
- フィード表示時に「プッシュで届いた分」+「フォロー中の有名人の最新」をマージ

**これが「有名人問題 (celebrity problem)」の標準解**。トレードオフを使い分ける典型例だ。

### データモデルと構成

```
posts(post_id, user_id, content, created_at)
follows(follower_id, followee_id)
feed_cache: Redis の各ユーザーごとのリスト(投稿IDを時系列で保持)
```

```
投稿 → アプリ → ① posts に保存
                ② fan-out サービスがフォロワーの feed_cache に push(一般ユーザー)
読み → アプリ → feed_cache を読む + フォロー中の有名人の最新を pull → マージ → 投稿本体を取得
```

### ボトルネックと対策

- **fan-out の負荷**: 非同期キュー(World 8 のメッセージング)でバックグラウンド配布。投稿のレスポンスは即返す
- **ストレージ**: feed_cache は投稿IDのみ(本体は posts から後で引く)。古い投稿は期限切れで削除
- **ホットな投稿**: CDN とキャッシュの多層化

### トレードオフ

- 一貫性は **結果整合で十分**(フィードに数秒遅れて出ても誰も困らない — GRAD 1 の一貫性モデルの選択)。可用性とスループットを優先する **AP 設計**

**面接での一言**: 「一般ユーザーはプッシュで読みを速く、有名人はプル型にして書き込み爆発を避ける、というハイブリッドが定石です。一貫性は結果整合で割り切ります。」

## 問題 4: チャットシステム(LINE/Slack 風)🔴

「1 対 1 とグループのリアルタイムチャット」を設計せよ。

### 要件と核心

- **機能**: メッセージ送受信、オンライン状態、既読、履歴
- **核心**: サーバから**クライアントへリアルタイムに push** する必要がある(HTTP は本来クライアント発)

### リアルタイム配信の選択肢

| 方式 | 仕組み | 評価 |
|---|---|---|
| ポーリング | 定期的に「新着ある?」と聞く | 単純だが無駄が多く遅延も大きい |
| ロングポーリング | 新着が出るまでサーバが応答を保留 | 改善版 |
| **WebSocket** | 双方向の常時接続を張る | チャットの定番。低遅延 |

**採用: WebSocket**。クライアントは接続サーバと常時接続を維持する。

### 構成

```
クライアント ⇄ WebSocket ⇄ 接続サーバ(どのユーザーがどのサーバにいるか管理)
                              ↓ メッセージキュー
                          メッセージ保存(DB)+ 相手の接続サーバへ転送
```

- **接続管理**: 「ユーザーID → 接続中サーバ」の対応を Redis などで共有。相手がどのサーバにいても届けられる
- **メッセージ保存**: 書き込みが多く時系列で読む → **NoSQL(Cassandra など、書き込みスケールに強い)**が向く。`(chat_id, timestamp)` で並べる
- **オフライン**: 相手が未接続ならキューに溜め、再接続時に配信。プッシュ通知も併用

### データモデル

```
messages(chat_id, message_id, sender_id, content, created_at)  -- chat_id でパーティション
chat_members(chat_id, user_id)
```

### トレードオフ

- メッセージ順序は **チャットごとの順序保証**が要る(全体の実時間順までは不要 — GRAD 1)。`chat_id` 単位でシーケンス番号を振る
- 既読・配信状態は結果整合で可

## 問題 5: 分散ユニークID生成器(Snowflake)🟡

「分散環境で、衝突しない・おおむね時系列順な 64 ビット ID」を生成する仕組みを設計せよ。問題1〜4 すべてで部品として必要になる土台。

### なぜ難しい?

- 単一の DB 連番は**単一障害点**かつボトルネック
- UUID(乱数)は衝突しないが **128 ビットで大きく、時系列順でない**(DB インデックスの局所性が悪い — World 11)

### 解: Twitter Snowflake 方式

64 ビットを意味のある区画に分ける:

```
| 1 bit 未使用 | 41 bit タイムスタンプ(ミリ秒) | 10 bit マシンID | 12 bit 連番 |
```

- **タイムスタンプ**が上位 → ID が**おおむね時系列順**(DB インデックスに優しい)
- **マシンID**で各サーバが独立に発番 → 中央調整不要、衝突なし
- **連番**で同一ミリ秒内の複数発行に対応(1 ミリ秒で 4096 個まで)

```python
class Snowflake:
    def __init__(self, machine_id):
        self.machine_id = machine_id
        self.seq = 0
        self.last_ts = -1

    def next_id(self, now_ms):
        if now_ms == self.last_ts:
            self.seq = (self.seq + 1) & 0xFFF   # 12 bit でラップ
            if self.seq == 0:
                now_ms = self._wait_next_ms(now_ms)  # 使い切ったら次ミリ秒を待つ
        else:
            self.seq = 0
        self.last_ts = now_ms
        return (now_ms << 22) | (self.machine_id << 12) | self.seq
```

### トレードオフ

- 時計の巻き戻し(NTP 同期でサーバ時刻が戻る)に注意。戻りを検知したら発番を一時停止するなどの対策が要る —— **分散システムでは「時刻」すら信用しきれない**(GRAD 1 の教訓)

## 🏆 設計工房クリアの心得

システム設計に銀の弾丸はない。だが**思考の順序**と**定番のトレードオフ**は決まっている:

| 状況 | 定番の判断 |
|---|---|
| 読みが圧倒的に多い | キャッシュ(Redis)+ CDN を主役に |
| 書きが多い・スケールが要る | NoSQL、シャーディング、非同期キュー |
| リアルタイム push が要る | WebSocket |
| フィード/配信で有名人がいる | プッシュ/プルのハイブリッド |
| 一意キーが要る | 連番の Base62 / Snowflake |
| 厳密な一貫性が要る(金銭) | 強一貫性・分散トランザクション(CP) |
| 多少古くてもいい(いいね・フィード) | 結果整合(AP) |
| 全 API の流量を守りたい | トークンバケツ + Redis |

**最重要の心構え**: 面接官は「正解」を求めていない。**「なぜその選択をしたか、何を犠牲にしたか」を説明できるか**を見ている。常に「これは読み多寄りなので…」「ここは一貫性より可用性を取って…」と理由を声に出すこと。
