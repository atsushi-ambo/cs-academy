# 📡 World 8: 通信の空中都市 — ネットワーク

> 地下都市のエレベーターを昇りきると、雲の上に浮かぶ都市があった。
> 無数の伝書鳩(パケット)が空を行き交い、世界中の都市へ手紙を運んでいる。
> 鳩は道に迷うし、手紙は濡れて破れるし、届く順番も入れ替わる。
> それでも世界の通信が成り立っているのはなぜか —— その魔法の名は **プロトコル** という。

**この章で学ぶこと(大学講義でいう「コンピュータネットワーク入門」)**
- プロトコルと階層モデル
- IP・TCP・UDP の役割分担
- DNS、HTTP/HTTPS
- 「URL を打ってからページが出るまで」の全貌

## 📖 講義 1: 空中都市の郵便制度 — 階層モデル(10 XP)

ネットワークは**層(レイヤー)**の積み重ねでできている。各層は自分の仕事だけをこなし、
下の層の詳細を知らなくていい —— ソフトウェア設計の「抽象化」と同じ思想だ。

| 層(TCP/IP モデル) | 仕事 | 代表プロトコル | たとえ |
|---|---|---|---|
| アプリケーション層 | 手紙の中身の書式 | HTTP, DNS, SMTP | 手紙の文面・言語 |
| トランスポート層 | 宛先の**アプリ**まで確実に届ける | TCP, UDP | 書留 or 普通郵便 |
| インターネット層 | 都市から都市へ経路を選ぶ | IP | 郵便局の経路網 |
| リンク層 | 隣の中継地点まで運ぶ | Ethernet, Wi-Fi | トラックと配達員 |

送信時は上から下へ各層がヘッダを付けて包み(カプセル化)、受信側で下から上へ剥がしていく。

## 📖 講義 2: 伝書鳩の掟 — IP と UDP と TCP(10 XP)

### IP — 「届くかは保証しない。最善は尽くす」

各マシンに **IP アドレス**を割り当て、パケットを宛先へ**ベストエフォート**で転送する。
途中で消えても、順番が入れ替わっても、重複しても知らんぷり。それが IP の流儀。

### UDP — 「とにかく速く投げる」

IP にポート番号(どのアプリ宛か)を足しただけの薄いプロトコル。
保証なし・コネクションなし・爆速。**ゲーム、通話、動画配信、DNS** など「多少欠けても遅延が嫌」な用途に。

### TCP — 「全部、順番どおり、確実に」

信頼できない IP の上に**信頼性を構築する**、このワールド最大の魔法。

- **3 ウェイハンドシェイク**で接続を確立: `SYN →` `← SYN+ACK` `ACK →`(互いに番号を同期)
- **シーケンス番号**で並べ直し、**ACK(確認応答)** が来なければ**再送**
- **フロー制御**: 受信側の処理能力に合わせて送る量を調整
- **輻輳制御**: ネットワークの混雑を察して送信ペースを落とす(混んだ高速道路に車を流し込まない)

| | TCP | UDP |
|---|---|---|
| 到達保証・順序保証 | ✓ | ✗ |
| 接続確立 | 必要(ハンドシェイク) | 不要 |
| 速度・オーバーヘッド | 重め | 軽い |
| 用途 | Web, メール, ファイル転送 | ゲーム, 通話, DNS |

## 📖 講義 3: 名前の魔法 — DNS(10 XP)

人間は `example.com` を覚え、機械は IP アドレスで通信する。**DNS** はその対応表を引く分散データベースだ。

```
ブラウザ → OS のキャッシュ → リゾルバ(ISP等)
            ↓ なければ階層をたどる
       ルートサーバ → .com サーバ → example.com の権威サーバ → IP アドレス!
```

- 木構造の**分散**システム(1 か所が全世界を知っているわけではない)
- 各段に**キャッシュ**があり(TTL 付き)、ほとんどの問い合わせは途中で即答される
- World 3 の木構造と World 7 のキャッシュの実戦例がこんな所に

## 📖 講義 4: 手紙の書式 — HTTP と HTTPS(10 XP)

**HTTP** は Web の手紙の書式。リクエストとレスポンスのシンプルな往復だ。

```
GET /index.html HTTP/1.1          HTTP/1.1 200 OK
Host: example.com         →      Content-Type: text/html
                          ←
                                  <html>...
```

- **メソッド**: GET(取得)、POST(送信)、PUT(更新)、DELETE(削除)
- **ステータスコード**: 200(成功)、301(引っ越し)、404(見つからない)、500(サーバ内部エラー)
- HTTP は**ステートレス**(毎回の手紙は独立)。状態は Cookie やトークンで持つ

**HTTPS** = HTTP + **TLS**。通信を暗号化し、相手が本物か**証明書**で確認する。
公開鍵暗号で鍵を交換し、以降は高速な共通鍵暗号で会話する(暗号の詳細は隠しダンジョンで)。

## ⚔️ クエスト

### クエスト 1: 鳩の使い分け(20 XP)

TCP と UDP のどちらを使うべきか、理由とともに答えよ。

1. オンライン対戦ゲームの位置情報の同期(毎秒 60 回送る)
2. 銀行の送金 API
3. ライブ動画配信
4. ファイルのダウンロード

<details>
<summary>📜 答え</summary>

1. **UDP** — 古い位置情報を再送してもらっても無意味。遅延最優先
2. **TCP** — 1 バイトも欠けてはならない
3. **UDP**(系) — 一瞬の欠けより遅延の少なさが大事(実際は QUIC や RTP などを利用)
4. **TCP** — 完全性が必須
</details>

### クエスト 2: 🔬 観測クエスト「鳩を目で見る」(30 XP)

実際に手を動かして観測せよ(macOS / Linux のターミナルで):

1. `dig example.com`(または `nslookup`)で DNS の回答と TTL を観察する
2. `curl -v https://example.com` で HTTP リクエスト/レスポンスヘッダと TLS ハンドシェイクのログを読む
3. `traceroute example.com` でパケットが経由するルーターの旅路を眺める

それぞれの出力のうち、講義に登場した概念(TTL、ステータスコード、経路)を 1 つずつ指差し確認すること。

### クエスト 3: 王道の問答「URL を打ってから何が起きる?」(30 XP)

`https://example.com` を打って Enter を押してからページが表示されるまでに起きることを、
**順番に、できるだけ多くの層に触れながら**説明せよ(これは実際の面接の超頻出問題だ)。

<details>
<summary>📜 模範解答の骨子</summary>

1. ブラウザが URL を解析(スキーム https、ホスト example.com、ポート 443)
2. **DNS 解決**: キャッシュ確認 → リゾルバ → ルート/.com/権威サーバの順で IP を得る
3. **TCP 接続**: 3 ウェイハンドシェイク(SYN / SYN+ACK / ACK)
4. **TLS ハンドシェイク**: 証明書検証と鍵交換で暗号化路を確立
5. **HTTP リクエスト送信**: `GET / HTTP/1.1`(以降の各パケットは IP がルーティングし、TCP が順序と再送を保証)
6. サーバが **HTTP レスポンス**(200 OK + HTML)を返す
7. ブラウザが HTML を解析、参照される CSS/JS/画像を並行取得しつつレンダリング
</details>

## 👹 ボス戦: 空中都市の門番「プロトコル・グリフォン」(100 XP)

ノートを閉じて挑め。7 割以上で勝利。

1. ネットワークが階層モデルで設計されている利点を、ソフトウェア設計の言葉で説明せよ。
2. TCP の 3 ウェイハンドシェイクの 3 つのパケットと、それぞれの役割を述べよ。
3. IP だけでは信頼できない通信を、TCP がどんな仕組みで信頼できるものにしているか 3 つ挙げよ。
4. DNS の問い合わせが世界中で高速に捌ける 2 つの設計上の理由は?
5. HTTP がステートレスであるとはどういう意味か。ログイン状態はどう実現する?
6. HTTPS が守ってくれるもの 2 つと、守ってくれないもの 1 つを挙げよ。

<details>
<summary>📜 解答</summary>

1. 各層がインターフェースだけで結ばれた抽象化(関心の分離)。下位層の実装(Wi-Fi か有線か)を入れ替えても上位層は変更不要。
2. SYN(接続要求+自分の初期シーケンス番号)、SYN+ACK(承諾+相手の番号確認+自分の番号)、ACK(確認)。双方の番号同期が完了して接続成立。
3. シーケンス番号による順序の復元、ACK とタイムアウトによる再送、チェックサムによる破損検出(+フロー制御・輻輳制御)。
4. 階層的な分散構造(負荷が根に集中しない)と、各段の TTL 付きキャッシュ。
5. 各リクエストが独立で、サーバは前のリクエストを覚えない。Cookie(セッション ID)やトークンをリクエストに載せて状態を擬似的に維持する。
6. 守る: 通信内容の機密性(暗号化)、通信相手の真正性(証明書)・改ざん検出。守らない: 例)アクセス先のドメイン名・通信量・タイミングは観測されうる(またはサーバ自体が悪意を持つ場合)。
</details>
