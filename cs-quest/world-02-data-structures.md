## 📖 講義 1: 武器屋の倉庫 — 配列(10 XP)

**配列**はメモリ上に**連続して**並んだ同じ型の要素の列。連続しているからこそ、
`先頭アドレス + 添字 × 要素サイズ` の計算 1 回で任意の要素に届く → **添字アクセス O(1)**。

**動的配列**(Python の list、Java の ArrayList、C++ の vector)は、固定長配列を内部に持ち、
一杯になったら **2 倍の容量を確保してコピー**する。World 1 で学んだとおり append は**償却 O(1)**。

| 操作 | 計算量 | メモ |
|---|---|---|
| 添字アクセス / 更新 | O(1) | 配列最大の武器 |
| 末尾に追加 (append) | O(1) 償却 | たまに拡張コピー |
| 先頭・中間に挿入 / 削除 | O(n) | 後ろの要素を全部ずらす |
| 検索(未ソート) | O(n) | 全部見るしかない |

## 📖 講義 2: 鍛冶屋の鎖 — 連結リスト(10 XP)

**連結リスト**は「値 + 次のノードへのポインタ」をつないだ鎖。メモリ上の連続性を捨てる代わりに、
**つなぎ替えだけで挿入・削除が O(1)** でできる(ただしその場所までの移動は O(n))。

```
[head] → [3|•] → [7|•] → [1|•] → None
```

| 操作 | 単方向リスト | 配列との比較 |
|---|---|---|
| 先頭に挿入 / 削除 | O(1) | 配列は O(n) |
| 添字アクセス | O(n) | 配列は O(1) |
| 検索 | O(n) | 同じ |
| ノードが分かっている時の削除 | O(1)(前ノード要) | 配列は O(n) |

**双方向リスト**は前ポインタも持ち、後ろ向きにも歩ける。LRU キャッシュの部品として頻出。

**ポインタ操作の心得**: 挿入・削除は「新ノードの next を先につなぎ、その後で前のノードの next を付け替える」。
順番を間違えると鎖の先が虚空に消える(=参照を失う)。紙に図を描いて修行せよ。

📺 [CS50 - Linked Lists(動画)](https://www.youtube.com/watch?v=2T-A_GFuoTo)

## 📖 講義 3: 皿洗い屋 — スタック(10 XP)

**スタック**は **LIFO**(Last-In, First-Out / 後入れ先出し)。皿の山と同じで、
最後に置いた皿を最初に取る。操作は `push`(積む)、`pop`(取る)、`peek`(覗く)— すべて **O(1)**。

**どこで使われる?**
- 関数呼び出し(コールスタック)— 再帰はスタックそのもの
- 括弧の対応チェック、構文解析
- ブラウザの「戻る」、エディタの Undo
- DFS(深さ優先探索)

## 📖 講義 4: 行列のできるパン屋 — キュー(10 XP)

**キュー**は **FIFO**(First-In, First-Out / 先入れ先出し)。並んだ順に買える行列。
`enqueue`(並ぶ)、`dequeue`(先頭が出る)— 連結リストか環状バッファで実装すれば **O(1)**。

⚠️ **罠**: 配列の先頭から `pop(0)` するとずらしが起きて O(n)。環状バッファ(リングバッファ)は
固定長配列に head/tail の添字を持ち、端まで来たら剰余で先頭に巻き戻すことで O(1) を保つ。

**どこで使われる?** BFS(幅優先探索)、タスクキュー、プリンタの待ち行列、メッセージング。

## 📖 講義 5: 錬金術師ハッシュの工房(10 XP)

町の最深部。**ハッシュテーブル**は「キー → 値」の対応を**平均 O(1)** で出し入れする魔法の棚だ。

**仕組み**:
1. **ハッシュ関数** h がキーを整数に変える: h("sword") = 982134…
2. 棚の数 m で割った余りでバケツを決める: 982134 % 8 = 6 → 6 番の棚へ
3. 取り出すときも同じ計算をするだけ → 平均 O(1)

**衝突(別のキーが同じ棚になる)の解決法**:
- **チェイン法**: 各棚を連結リストにしてぶら下げる
- **オープンアドレス法(線形探査)**: 埋まっていたら隣の棚を順に探す

**負荷率** α = 要素数 / 棚の数。α が大きくなると衝突が増えて性能が落ちるため、
閾値(例: 0.75)を超えたら**棚を倍に増やして全要素を入れ直す**(リハッシュ)。これも償却 O(1)。

| 操作 | 平均 | 最悪 |
|---|---|---|
| 挿入 / 検索 / 削除 | O(1) | O(n)(全部衝突した場合) |

**面接での合言葉**: 「重複チェック」「カウント」「対応表」と聞こえたら、まずハッシュテーブルを疑え。

### クエスト 2: 💻 実装クエスト「動的配列を鍛える」(50 XP)

組み込みの可変長リストを使わずに、固定長領域(または固定長配列のシミュレーション)だけで
動的配列を実装せよ。必要なメソッド:

- `size()` / `capacity()` / `is_empty()`
- `at(index)` — 範囲外なら明確に失敗させる
- `push(item)`(満杯なら容量 2 倍に resize)
- `insert(index, item)` / `prepend(item)`
- `pop()` / `delete(index)` / `remove(item)` / `find(item)`
- 縮小: サイズが容量の 1/4 になったら容量を半分に

### クエスト 4: 💻 実装クエスト「ハッシュの工房に弟子入り」(50 XP)

オープンアドレス法(線形探査)でハッシュテーブルを実装せよ:
`hash(k, m)`, `add(key, value)`(既存キーは上書き), `exists(key)`, `get(key)`, `remove(key)`。
⚠️ remove は「削除済みマーカー(墓石)」を置かないと、後続の探査が途切れるぞ。

### クエスト 5: 括弧の番人(20 XP)

`"([{}])"` のような文字列が与えられる。括弧の対応が正しいか O(n) で判定するアルゴリズムを書け。

<details>
<summary>📜 ヒント → 答え</summary>

開き括弧をスタックに push。閉じ括弧が来たら pop して種類が一致するか確認。
最後にスタックが空なら正しい。途中で不一致・空 pop が起きたら不正。
</details>

## 👹 ボス戦: 城下町の関所役人「五構造の試し」(100 XP)

ノートを閉じて挑め。7 割以上で勝利。

1. 動的配列の append が「償却 O(1)」である理由を 2 文で説明せよ。
2. 配列と連結リスト、それぞれが勝つ場面を 1 つずつ挙げよ。
3. キューを 2 つのスタックで実装する方法を説明せよ。enqueue / dequeue の償却計算量は?
4. ハッシュテーブルの負荷率とは何か。なぜリサイズが必要か。
5. ハッシュテーブルの最悪計算量が O(n) になるのはどんなときか。
6. 連結リストに「途中から輪っかになっている」(循環がある)かを O(1) の追加メモリで検出する方法は?

<details>
<summary>📜 解答</summary>

1. 容量拡張のコピー総数は等比数列の和で高々 2n。n 回の操作にならすと 1 回あたり O(1)。
2. 配列: 添字アクセスが多い場面(O(1))。連結リスト: 先頭への挿入・削除が多い場面(O(1))。
3. 入力用スタック A と出力用スタック B。enqueue は A に push。dequeue は B が空なら A を全部 B に移してから pop(順序が反転して FIFO になる)。各要素は高々 2 回しか移動しないので償却 O(1)。
4. 負荷率 = 要素数 ÷ バケツ数。大きくなると衝突が増え O(1) が崩れるため、閾値超過で棚を増やして入れ直す。
5. ハッシュ関数が偏って全キーが同じバケツに入ったとき(チェインが 1 本の長いリストになる)。
6. **フロイドの循環検出(うさぎとかめ)**: 1 歩ずつ進むポインタと 2 歩ずつ進むポインタを走らせ、追いつけば循環あり。
</details>
