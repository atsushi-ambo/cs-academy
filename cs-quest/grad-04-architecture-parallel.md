## 📖 講義 1: 機械の母語 — ISA とパイプライン(20 XP)

**命令セットアーキテクチャ (ISA)** は、ハードとソフトの**契約書**。
CPU が理解する命令の語彙(add, load, branch...)を定義する。
- **RISC**(ARM, RISC-V): 単純な命令を高速に。スマホ・現代の主流
- **CISC**(x86): 複雑な命令も持つ。歴史的経緯と互換性

### パイプライン — 流れ作業で命令を捌く
洗濯にたとえる: 洗う→乾かす→畳む を 1 着ずつやらず、洗濯機が次を洗う間に前のを乾燥機へ。
古典的 5 段: **フェッチ → デコード → 実行 → メモリ → ライトバック**(World 7 の 4 拍子の精密版)。
理想は「1 サイクルに 1 命令完了」。だが現実には**ハザード**が流れを乱す。

## 📖 講義 2: 流れを乱す者 — ハザードと投機(20 XP)

| ハザード | 原因 | 対策 |
|---|---|---|
| 構造ハザード | 同じ装置を 2 命令が同時に欲しがる | 装置を増やす |
| データハザード | 前の命令の結果を次が待つ | フォワーディング(結果を計算後すぐ横流し)、ストール |
| **制御ハザード** | 分岐の行き先が決まるまで次が読めない | **分岐予測** |

### 分岐予測と投機実行
CPU は「この if はたぶん成立する」と**予測して先に実行**してしまう。当たれば爆速、外れたら捨てて巻き戻す。
現代 CPU の予測的中率は 95% 超 —— だが、この投機こそが **Spectre/Meltdown**(投機実行の痕跡から秘密を盗む)
という GRAD 2 のセキュリティ問題を生んだ。**抽象は漏れる**(leaky abstraction)の生きた教訓だ。

**実務への接続**: なぜ「ソート済み配列のループが速い」ことがあるのか? → 分岐予測が当たりやすいから。
World 7 のキャッシュと合わせ、「同じ O(n) でも実測が変わる」理由の核心がここにある。

## 📖 講義 3: 記憶の真の深さ — キャッシュコヒーレンス(20 XP)

World 7 でキャッシュを学んだ。だが**コアが複数ある**と新たな悪夢が生まれる:
コア A とコア B が同じ変数を各自のキャッシュに持ち、A が書き換えたら B のコピーは**古い嘘**になる。

- **キャッシュコヒーレンスプロトコル(MESI 等)**: 各キャッシュラインに状態(Modified/Exclusive/Shared/Invalid)を持たせ、
  誰かが書いたら他のコピーを無効化する。ハードウェアが裏で必死に整合性を保っている
- **フォルスシェアリング**: 別々の変数でも同じキャッシュライン(64 バイト)に同居すると、
  無関係なのに無効化が飛び交って激遅になる。並列プログラムの隠れた性能killer
- **メモリ一貫性モデル**: マルチコアでの読み書きの見え方の規則。GRAD 1 の分散一貫性が、1 チップの中にもある

## 📖 講義 4: 速度の終焉と並列の夜明け — アムダールの法則(20 XP)

かつて CPU は年々クロックが上がった。だが**電力と発熱の壁**(デナードスケーリングの終焉)で
「1 コアを速く」が限界に。以降の進歩は**コアを増やす(並列)**方向へ。プログラマの時代が変わった瞬間だ。

### アムダールの法則 — 並列化の冷徹な天井
プログラムの割合 p だけが並列化可能で、n 個のコアを使うとき、速度向上は:

```
高速化 = 1 / ((1−p) + p/n)
```

p = 0.95(95% 並列化可能)でも、n → ∞ で高速化は **最大 20 倍**で頭打ち。
**逐次部分(5%)が全体を縛る**。「コアを 1000 個積めば 1000 倍」は幻想。
—— World 1 の漸近解析が、並列の世界では「並列化できない部分」の解析として蘇る。

### 並列のパターン(World 14 のアルゴリズム設計の並列版)
- **データ並列**: 同じ処理を多数のデータに(map。GPU の得意技)
- **タスク並列**: 異なる処理を同時に
- **分割統治の並列化**: マージソートの左右を別コアで(World 4 + World 14)
- **同期の罠**: ロック競合・偽共有・負荷の偏りが理論値を奪う(GRAD 1/World 10 の並行性が物理で再来)

## 📖 講義 5: 万の小人の軍勢 — GPU とアクセラレータ(20 XP)

CPU が「少数の賢い巨人」なら、GPU は「**数千の単純な小人**」。
同じ命令を大量のデータに一斉に適用する(SIMT)のが恐ろしく得意。

- 行列積・畳み込み —— GRAD 3 の深層学習の計算は、まさに GPU の独壇場(だから AI ブームと GPU は不可分)
- **メモリ帯域**が律速になりがち: 計算より「データを運ぶ」のが遅い(World 7 の局所性が桁違いの規模で効く)
- TPU など専用アクセラレータは「特定の計算(テンソル積)だけを爆速に」特化したもの

**最終講義の締め**: 抽象(高級言語)から物理(トランジスタと電力)まで降りてきた君は、
もう「コードがなぜ速い/遅いか」を**全レイヤーで説明できる**。それが計算機科学者の視座だ。

## ⚔️ クエスト

### クエスト 1: パイプラインの目(40 XP)

1. 5 段パイプラインが理想的に動けば、N 命令の実行サイクル数はおよそいくつか(素朴な非パイプラインと比較)。
2. データハザードを「フォワーディング」がどう解決するか説明せよ。
3. 分岐予測が外れたときに失われるものは何か。なぜループは予測しやすいか。

<details>
<summary>📜 答え</summary>

1. 非パイプライン ≈ 5N サイクル。理想パイプラインは初期の充填後は 1 サイクル/命令なので ≈ N + 4(パイプライン段数−1)。N が大きいと約 5 倍速い。
2. 前の命令の計算結果を、レジスタへの書き戻しを待たずに、後続命令の実行ステージへ直接横流しすることでストールを減らす。
3. 投機的に実行した命令を破棄してパイプラインを流し直すサイクル(分岐ミスペナルティ)。ループは大半の回で「継続(成立)」するため、同じ予測が当たり続けやすい。
</details>

### クエスト 2: 💻 実装クエスト「並列化の天井を測る」(80 XP)

大きな配列の総和(または行列積)を、1 スレッドと N スレッドで実装し実測せよ:

1. スレッド数 1, 2, 4, 8 での実行時間を計測し、高速化率をグラフ化
2. 理想の線形高速化から外れる点を観察し、アムダールの法則・同期コスト・メモリ帯域で考察
3. 余力があれば、同じ変数を複数スレッドで更新する版を書き、**偽共有**や競合で逆に遅くなる現象を再現

### クエスト 3: アムダールの計算(40 XP)

1. プログラムの 80% が並列化可能。4 コア・無限コアでの最大高速化は?
2. 「並列化率を 80% → 95% に上げる」のと「コアを 4 → 16 に増やす」の、どちらが高速化に効くか(数値で比較)。
3. この結果が「最適化はまず逐次ボトルネックを潰せ」という教訓につながる理由は?

<details>
<summary>📜 答え</summary>

1. 4 コア: 1/(0.2 + 0.8/4) = 1/0.4 = **2.5 倍**。無限コア: 1/0.2 = **5 倍**が上限。
2. p=0.8,n=16: 1/(0.2+0.05)=4 倍。p=0.95,n=4: 1/(0.05+0.2375)=約3.4倍。p=0.95,n=16: 1/(0.05+0.059)=約9.2倍。並列化率を上げる方が天井(1/(1−p))自体を押し上げるため効果が大きい。
3. 逐次部分 (1−p) が高速化の上限 1/(1−p) を決めるため、コアを増やすより逐次ボトルネックを減らす方が根本的に効く。
</details>

### クエスト 4: 物理の問答(40 XP)

1. なぜ CPU のクロック競争が止まり、マルチコアへ移行したのか。
2. キャッシュコヒーレンスとは何を保証する仕組みか。フォルスシェアリングはなぜ起きるか。
3. 深層学習に GPU が向く理由を、計算の性質から説明せよ。

<details>
<summary>📜 答え</summary>

1. クロックを上げると消費電力と発熱が急増し(デナードスケーリングの終焉)、冷却と電力の物理的限界に達したため。性能向上をコア数(並列)で稼ぐ方向に転換した。
2. 複数コアのキャッシュにある同一メモリの複製を整合させ、あるコアの書き込みが他コアに正しく反映される(古い値を読まない)ことを保証する。フォルスシェアリングは、論理的に無関係な変数が同一キャッシュライン(64B)に同居し、片方の更新でライン全体が無効化されることで起きる。
3. 深層学習の中心は行列積・畳み込みで、同一演算を大量データに独立並列適用するデータ並列性が高い。GPU は数千の単純コアで同じ命令を一斉実行(SIMT)するのに最適。
</details>

## 👹 ボス戦: 万物の源「アーキテクト・プライム」(200 XP)

ノートを閉じて挑め。7 割以上で勝利。**これは大学院篇 最後のボスだ。**

1. ISA が「ハードとソフトの契約」と呼ばれる意味を説明せよ。RISC と CISC の設計思想の違いは?
2. パイプラインの 3 種のハザードを挙げ、それぞれの対策を述べよ。
3. 分岐予測の投機実行が、性能とセキュリティの両面に与えた影響を述べよ(Spectre に触れて)。
4. マルチコアにおけるキャッシュコヒーレンスの必要性と、それが分散システム(GRAD 1)の一貫性問題とどう似ているか述べよ。
5. アムダールの法則を述べ、「コアを増やせばいくらでも速くなる」という主張を定量的に反駁せよ。
6. 同じ O(n) のアルゴリズムでも実測速度が大きく違いうる理由を、キャッシュ・分岐予測・並列性の 3 観点から説明せよ。
7. 抽象化の階層(高級言語 → コンパイラ → ISA → マイクロアーキテクチャ → 回路)を辿り、各層が下層をどう隠蔽し、どこで抽象が漏れるか 1 例ずつ挙げよ。

<details>
<summary>📜 解答の骨子</summary>

1. ISA はソフトが依存してよい命令・レジスタ・メモリモデルの仕様で、これを守る限り異なる実装(マイクロアーキテクチャ)でも同じプログラムが動く。RISC は単純命令を高速・低電力に、CISC は複雑命令と後方互換を重視。
2. 構造(資源競合→資源増設)、データ(依存→フォワーディング/ストール)、制御(分岐→分岐予測/遅延スロット)。
3. 性能: 分岐待ちのストールを投機実行で隠し大幅高速化。セキュリティ: 投機的に実行された命令がキャッシュに残す痕跡から、本来アクセスできない秘密を推定できる(Spectre/Meltdown)。抽象(投機は見えないはず)が漏れた例。
4. 複数キャッシュの同一データの複製を整合させる必要があり、これは複数ノードのデータ複製を一貫させる分散システムの問題と本質的に同型(無効化/更新の伝播、一貫性モデル)。スケールが 1 チップ内か地理分散かが違うだけ。
5. 高速化 = 1/((1−p)+p/n)。逐次部分 (1−p) が残るため n→∞ でも上限は 1/(1−p)。例: p=0.95 なら最大 20 倍で、コアをいくら増やしても超えられない。
6. キャッシュ: 連続アクセスでヒット率が上がると数倍速い。分岐予測: 予測しやすいパターンだとストールが減る。並列性: データ並列に分解できれば複数コアで実時間短縮。いずれも漸近計算量には現れない定数因子を左右する。
7. 高級言語(メモリ管理を隠す→GC のポーズで漏れる)、コンパイラ(命令選択を隠す→最適化で性能が変わり漏れる)、ISA(実装を隠す→投機実行のサイドチャネルで漏れる)、マイクロアーキ(タイミングを隠す→キャッシュタイミングで漏れる)、回路(物理を隠す→消費電力解析で漏れる)。
</details>
