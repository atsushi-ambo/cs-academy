## 📖 講義 1: 神殿の二重の扉 — カーネルとシステムコール(15 XP)

OS の仕事は 3 つ: **仮想化**(各プログラムに「マシンを独り占めしている」と錯覚させる)、
**並行管理**、**永続化**(ファイル)。

CPU には **ユーザモード** と **カーネルモード** の 2 つの特権レベルがある。
アプリは檻(ユーザモード)の中で走り、ハードウェアに触りたいときは**システムコール**で神殿に祈る:

```
アプリ: read(fd, buf, 100) を呼ぶ
  → トラップ命令で CPU がカーネルモードへ遷移
  → カーネルが検査・実行(ディスクから読む)
  → 結果を返してユーザモードに復帰
```

`open / read / write / fork / exec / wait / exit` —— UNIX の祈りの言葉たち。
**割り込み(interrupt)** はハードウェアからの神託(キー入力、タイマー、ディスク完了)で、
これもカーネルが受け取る。タイマー割り込みこそが「OS がアプリから CPU を取り上げられる」力の源泉だ。

## 📖 講義 2: 魂の輪廻 — プロセスとスケジューリング(15 XP)

**プロセスの一生**: 生成(fork)→ 実行可能(ready)⇄ 実行中(running)⇄ 待機(blocked)→ 終了(zombie → 回収)

UNIX の出生の秘儀:
- **fork()** — 自分の完全な複製を作る(親には子の PID、子には 0 が返る)
- **exec()** — 自分の中身を別のプログラムに入れ替える
- シェルがコマンドを実行する正体は **fork して子が exec し、親が wait する**

### スケジューリング — 誰に CPU を与えるか

| 方式 | 仕組み | 長所 / 短所 |
|---|---|---|
| FIFO | 来た順 | 単純 / 長い仕事の後ろで皆が待つ(コンボイ効果) |
| SJF(最短ジョブ優先) | 短い仕事から | 平均待ち時間最小 / 未来の長さは分からない |
| ラウンドロビン | タイムスライスで交代 | 応答性が良い / 切替オーバーヘッド |
| **MLFQ(多段フィードバック)** | 短い対話的ジョブを自動で優先、CPU を使い倒すジョブは降格 | 実用 OS の原型。過去の挙動から未来を予測する |

## 📖 講義 3: 同時に祈る者たち — 並行性の本殿(15 XP)

World 7 でデータ競合とデッドロックを見た。神殿では正式な道具を授ける。

- **ロック(mutex)**: 臨界区間に入れるのは 1 人。実装はハードウェアのアトミック命令(test-and-set / compare-and-swap)に根ざす
- **条件変数**: 「ある条件が成り立つまで眠り、成り立ったら起こしてもらう」。
  `wait(cond, lock)` は**ロックを手放して眠り、起きたら取り直す**。
  ⚠️ 起きたら **while で条件を再確認**(spurious wakeup と横取りがあるから `if` では祈りが足りない)
- **セマフォ**: 整数カウンタ + 待ち行列。初期値 1 ならロック、初期値 n なら「n 席の入場制限」

### 名物修行: 生産者・消費者問題

固定長バッファに、生産者がアイテムを入れ、消費者が取り出す。
「満杯なら生産者が待つ」「空なら消費者が待つ」を条件変数 2 つ(または セマフォ empty/full)で書く。
**これが書ければ並行プログラミングの初段**。メッセージキュー、スレッドプール、パイプラインの原型だ。

## 📖 講義 4: 神殿最大の幻術 — 仮想メモリ(15 XP)

各プロセスは「広大なメモリを独り占めしている」と思っている。幻である。

- **仮想アドレス → 物理アドレス** の変換を、**ページ**(通常 4 KB)単位で行う
- 変換表が**ページテーブル**(プロセスごとに 1 つ。多段構造で省メモリ化)
- 毎回表を引いたら遅すぎるので、変換結果を **TLB** という専用キャッシュに置く(World 7 の局所性、再び)

**ページフォールト** — アクセスしたページが物理メモリにないと CPU が例外を上げ、カーネルが登場:
- 正当なページがディスク(スワップ)にある → 読み込んで再実行(**デマンドページング**)
- 不正なアクセス → **segmentation fault**(君が C で見たあれの正体)

物理メモリが足りないときは**ページ置換**(LRU 近似が定番)。
置換が追いつかずディスク往復だらけになる地獄を**スラッシング**と呼ぶ。

**この仕組みの恩恵**: プロセス間の隔離(他人のメモリは見えない)、fork の copy-on-write、
メモリマップトファイル、共有ライブラリ —— 現代 OS の魔法はほぼ全部ページングの応用だ。

## 📖 講義 5: 永遠の書庫 — ファイルシステム(15 XP)

- **inode**: ファイルの本体(メタデータ + データブロックの位置表)。**ファイル名は inode への参照にすぎない**(ハードリンクが作れる理由)
- **ディレクトリ**: 「名前 → inode 番号」の表が入った特殊なファイル
- パス解決 `/home/user/quest.txt` = ルート inode から表を 1 段ずつ引く旅

### クラッシュ一貫性 — 書き込み中に電源が落ちたら?

ファイル追記は「データブロック書き込み + inode 更新 + 空きビットマップ更新」の複数操作。
途中で落ちると矛盾した書庫が残る。現代の解は **ジャーナリング(WAL)**:

> **「これからやる変更を先にログへ書き、コミット印を打ってから本番に反映する」**
> 落ちたら再起動時にログを見て、コミット済みはやり直し、未コミットは捨てる。

この「**先にログを書く**」という思想は、World 11 のデータベース(WAL/ARIES)、
大学院篇の分散システム(Raft のログ複製)で**何度でも再登場する、CS 全体の貫通テーマ**だ。

## ⚔️ クエスト

### クエスト 1: 神官の問答(30 XP)

1. `fork()` を呼んだ直後、`printf` が 2 回表示されるのはなぜか。
2. タイマー割り込みがないと、OS はなぜ無限ループするプロセスを止められないのか。
3. 「ゾンビプロセス」とは何で、なぜ存在する必要があるのか。

<details>
<summary>📜 答え</summary>

1. fork は呼び出し時点の完全な複製を作り、親と子の**両方**が fork の次の行から実行を続けるから。
2. ユーザモードのプロセスが自発的にシステムコールを呼ばない限りカーネルに制御が戻らない。タイマー割り込みは定期的に強制的にカーネルへ制御を戻す唯一の保証。
3. 終了したが親がまだ wait() で終了コードを回収していないプロセス。終了コードを親に届けるまで PID とステータスを保持しておく必要がある。
</details>

### クエスト 2: 💻 実装クエスト「生産者・消費者の儀式」(70 XP)

手元の言語のスレッドとロック・条件変数(Python なら `threading`、Java なら `synchronized`/`Condition`)で、
固定長バッファの生産者・消費者を実装せよ。

- 生産者 2、消費者 2、バッファ長 5 で、アイテム 1000 個が**欠けも重複もなく**流れることを確認
- `wait` を `if` で書いた場合に何が起きうるか、コメントで説明を書くこと

### クエスト 3: スケジューラの机上演習(30 XP)

ジョブ A(実行 10ms, 到着 0ms)、B(実行 2ms, 到着 1ms)、C(実行 4ms, 到着 2ms)。
FIFO、SJF(到着済みから選ぶ)、RR(スライス 2ms)それぞれの完了順と平均ターンアラウンド時間を計算せよ。

<details>
<summary>📜 答え</summary>

- FIFO: A(10), B(12), C(16) → ターンアラウンド (10 + 11 + 14)/3 ≈ 11.7ms
- SJF(非横取り): A(10), B(12), C(16) は A 実行中に B,C 到着、A 完了後 B→C で同じ。横取り(STCF)なら B が割り込み: B(3), C(8?)…と改善する(各自タイムラインを描け)
- RR: A と B と C が 2ms ずつ交代。タイムラインを描き、対話的ジョブ(短い B)の応答が早いことを確認せよ
</details>

### クエスト 4: 幻術の見破り(30 XP)

1. 32 ビット仮想アドレス、ページサイズ 4 KB のとき、仮想アドレスは「ページ番号何ビット + オフセット何ビット」に分かれるか。
2. TLB ヒット率が 99% でもなお性能に効く理由を、World 7 のメモリ階層の言葉で説明せよ。
3. fork が高速な理由「copy-on-write」を説明せよ。

<details>
<summary>📜 答え</summary>

1. オフセット 12 ビット(2¹² = 4096)+ ページ番号 20 ビット。
2. TLB ミス 1 回はページテーブル参照=メモリアクセス数回分(多段なら段数分)のコスト。メモリはキャッシュより 1〜2 桁遅いため、1% のミスでも平均アクセス時間を大きく押し上げる。
3. fork 時はページテーブルだけ複製して物理ページを親子で共有し、全ページを読み取り専用にマークする。どちらかが**書いた瞬間に**そのページだけ複製する。ほとんど書かれないページのコピーを永遠に回避できる。
</details>

## 👹 ボス戦: 神殿の主「カーネル・セラフィム」(150 XP)

ノートを閉じて挑め。7 割以上で勝利。

1. ユーザモードとカーネルモードを分ける理由と、その境界を越える 2 つの経路(自発・強制)を述べよ。
2. シェルが `ls | grep txt` を実行するとき、fork / exec / pipe がどう使われるか説明せよ。
3. 条件変数の wait をなぜ while ループで囲むのか。
4. デマンドページングの流れを「アクセス → フォールト → 解決 → 再実行」の順で説明せよ。
5. MLFQ が「短い対話的ジョブ」を、実行時間を事前に知らずに優先できる仕組みは?
6. ジャーナリングファイルシステムがクラッシュ後に一貫性を回復できる理由を「先にログ」の原則で説明せよ。
7. スラッシングとは何か。検知したら OS は何をすべきか。

<details>
<summary>📜 解答の骨子</summary>

1. アプリの暴走・悪意からハードウェアと他プロセスを守る隔離のため。自発=システムコール(トラップ)、強制=割り込み・例外。
2. 親が pipe() でパイプを作り、fork を 2 回。子 1 は標準出力をパイプ書き口に付け替えて exec("ls")、子 2 は標準入力をパイプ読み口に付け替えて exec("grep")。親は両方を wait。
3. 起床時に条件が成り立つ保証がないから(偽の起床、他スレッドによる条件の横取り)。while なら取り直して再確認できる。
4. ページにアクセス → 存在ビットが無効で CPU が例外 → カーネルが正当性を確認し、空き枠を確保(必要なら LRU で追い出し)、ディスクから読み込み、ページテーブルを更新 → 命令を再実行。
5. 新規ジョブは最高優先度から始め、タイムスライスを使い切るたびに降格する。短い対話的ジョブは使い切る前に I/O 待ちに入るので高優先度に留まる。挙動の実績で分類する仕組み。
6. 変更内容を先にジャーナルへ書き、コミット記録を打ってから本体を更新する。クラッシュ時はジャーナルを走査し、コミット済みトランザクションのみ再適用(または未完了を巻き戻し)すれば、本体は常に「全部適用」か「全部未適用」のどちらかになる。
7. 物理メモリ不足でページの追い出しと読み込みが往復し、CPU がほぼ I/O 待ちになる状態。プロセス数を減らす(スワップアウト/admission control)かメモリを増やすしかない。
</details>
