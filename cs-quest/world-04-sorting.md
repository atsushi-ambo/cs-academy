# 🌋 World 4: 整列の火山 — ソートアルゴリズム

> 火山のふもとに、5 人の鍛冶師が住んでいる。
> それぞれ流儀の違う「整列の技」で剣を打つ。
> 弟子入りして全員の技を見極めた者だけが、火口に眠る伝説の剣
> **「エヌログエヌの剣」** を抜くことができる。

**この章で学ぶこと(大学講義でいう「アルゴリズム 第 3〜5 回」)**
- 比較ソート 5 種の仕組み・計算量・特性
- 安定性、in-place 性という比較軸
- 分割統治法という設計パラダイム
- 比較ソートの理論限界 Ω(n log n)

---

## 📖 講義 1: 弟子入り前の心得 — ソートを測る 3 つの目(10 XP)

| 観点 | 意味 |
|---|---|
| 時間計算量 | 最悪・平均でどれだけ比較・交換するか |
| 空間計算量 / in-place | 追加メモリ O(1) で済むか(in-place)、O(n) 必要か |
| **安定性 (stable)** | 同じキーの要素の**元の順序が保たれる**か |

安定性の例: 名簿を「名前順」でソート後に「学年順」で安定ソートすると、
**同学年の中では名前順が保たれる**。多段ソートの基礎であり、面接でもよく問われる。

## 📖 講義 2: 五人の鍛冶師(10 XP)

### 🔨 選択ソート — 「最小を選んで前へ」流

未ソート部分から最小値を**選び**、先頭と交換。これを繰り返す。
- 常に **O(n²)**。ただし**交換回数は最大 n−1 回**(書き込みが高価な場面では悪くない)
- in-place / 不安定

### 🔨 挿入ソート — 「手札に差し込む」流

トランプの手札を整えるように、各要素をソート済み部分の正しい位置に**挿入**する。
- 最悪 O(n²)、しかし**ほぼソート済みの入力なら O(n)**
- in-place / 安定
- **小さい配列では最速級** — 実用ソートの「仕上げ役」として今も現役

### 🔨 マージソート — 「半分に割って混ぜる」流(分割統治)

```
mergesort(A):
    if |A| ≤ 1: return A
    L = mergesort(前半)
    R = mergesort(後半)
    return merge(L, R)   # 2 つのソート済み列を O(n) で混ぜる
```

- **常に O(n log n)**(log n 段 × 各段 O(n) のマージ)
- 安定 / ただし O(n) の作業領域が必要
- 連結リストのソート、外部ソート(巨大ファイル)に強い

### 🔨 クイックソート — 「軸より小か大かで仕分ける」流(分割統治)

ピボット(軸)を 1 つ選び、**小さい組と大きい組に仕分け(パーティション)**、それぞれを再帰的にソート。
- 平均 **O(n log n)**、最悪 O(n²)(ピボットが毎回端の値になる場合 — 例: ソート済み入力で先頭をピボットに選ぶ)
- 対策: ランダムにピボットを選べば最悪ケースは事実上起きない
- in-place / 不安定
- **キャッシュ効率が良く、実測では最速級**(World 7 でその理由が分かる)

### 🔨 ヒープソート — 「ヒープから順に抜く」流

配列を heapify (O(n)) してから、最大値を末尾に置いて extract_max を n 回。
- **常に O(n log n)** / in-place / 不安定
- 追加メモリゼロで最悪 O(n log n) 保証が欲しいときの選択肢

### 📊 五人の技、総覧

| 技 | 最悪 | 平均 | 追加メモリ | 安定 | 一言 |
|---|---|---|---|---|---|
| 選択 | O(n²) | O(n²) | O(1) | ✗ | 交換が少ない |
| 挿入 | O(n²) | O(n²) | O(1) | ✓ | ほぼ整列済みなら O(n) |
| マージ | O(n log n) | O(n log n) | O(n) | ✓ | 常に安定の優等生 |
| クイック | O(n²) | O(n log n) | O(log n)※ | ✗ | 実測最速級 ※再帰スタック |
| ヒープ | O(n log n) | O(n log n) | O(1) | ✗ | 保証と省メモリ |

## 📖 講義 3: 火山の真実 — 比較ソートの限界(10 XP)

**定理**: 比較だけで並べ替えるソートは、最悪 **Ω(n log n)** 回の比較が必要。

**証明の直感**: n 要素の並び方は n! 通り。比較 1 回で可能性は高々半分にしかならない。
n! 通りを 1 通りに絞るには log₂(n!) ≈ n log n 回の比較が要る(決定木の高さの議論)。

ただしこれは「比較」の限界。**比較しないソート**は壁を越える:
- **カウンティングソート**: 値の範囲 k が小さければ O(n + k)
- **基数ソート**: 桁ごとに安定ソート、O(d·(n + k))

## ⚔️ クエスト

### クエスト 1: 鍛冶師の見極め(20 XP)

最適なソートを選び、理由を述べよ。

1. 1,000 万件のログを安定にソートしたい。メモリは十分ある
2. ほぼソート済みの配列に数件だけ新データが混ざっている
3. 組み込み機器。追加メモリほぼゼロで最悪 O(n log n) を保証したい
4. 0〜100 の整数 100 万個をソートしたい

<details>
<summary>📜 答え</summary>

1. **マージソート**(安定 + 常に O(n log n))
2. **挿入ソート**(ほぼ整列済みなら O(n))
3. **ヒープソート**(in-place + 最悪 O(n log n))
4. **カウンティングソート**(値域が小さい → O(n + k))
</details>

### クエスト 2: 手作業の修行(20 XP)

配列 `[5, 2, 9, 1, 7]` について:
1. 選択ソートの各パス後の配列を書け
2. マージソートの分割と統合の過程を木の図で書け
3. 先頭をピボットとするクイックソートの最初のパーティション結果を書け

<details>
<summary>📜 答え(例)</summary>

1. `[1,2,9,5,7]` → `[1,2,9,5,7]` → `[1,2,5,9,7]` → `[1,2,5,7,9]`
2. `[5,2,9,1,7]` → `[5,2]` `[9,1,7]` → … → 統合で `[2,5]` `[1,7,9]` → `[1,2,5,7,9]`
3. ピボット 5: 小さい組 `[2,1]` + 5 + 大きい組 `[9,7]`(実装により並びは異なって良い)
</details>

### クエスト 3: 💻 実装クエスト「二つの名剣」(50 XP × 2)

1. **マージソート**を実装せよ(再帰 + merge 関数)
2. **クイックソート**を実装せよ(in-place パーティション。ピボットはランダム選択)

両方とも、乱数配列・ソート済み配列・逆順配列・重複だらけの配列でテストすること。

### クエスト 4: 鍛冶師の問答(20 XP)

1. クイックソートが最悪 O(n²) になる入力と、その回避策は?
2. マージソートの再帰の深さは? 各段の仕事量は?
3. 「安定なソートが必要」な具体例を 1 つ挙げよ。

<details>
<summary>📜 答え</summary>

1. 既にソート済みの配列で常に端をピボットに選ぶ場合(分割が 1 : n−1 に偏る)。ランダムピボットや中央値の三点取りで回避。
2. 深さ O(log n)、各段の merge 合計は O(n)。よって O(n log n)。
3. 例: 取引履歴を金額でソートした後、日付で安定ソートして「同日内は金額順」を保つ。
</details>

## 👹 ボス戦: 火口の番竜「ソートドラゴン」(100 XP)

ノートを閉じて挑め。7 割以上で勝利。

1. 5 つの比較ソートの最悪計算量・安定性・in-place 性を表で書け(暗記で)。
2. 比較ソートが Ω(n log n) より速くなれない理由を決定木で説明せよ。
3. クイックソートのパーティション(Lomuto か Hoare のどちらか)を擬似コードで書け。
4. マージソートが連結リストと相性が良い理由は?
5. 実用言語のソート(Python の Timsort 等)が「マージ + 挿入」のハイブリッドである理由を述べよ。

<details>
<summary>📜 解答</summary>

1. 講義 2 の表を参照。
2. 比較ソートは比較結果で分岐する二分決定木とみなせる。葉は n! 通りの並べ替えに対応する必要があり、高さ(=最悪比較回数)は log₂(n!) = Ω(n log n)。
3. (Lomuto 例)pivot = A[hi]; i = lo−1; for j in lo..hi−1: if A[j] ≤ pivot: i++; swap(A[i], A[j]); 最後に swap(A[i+1], A[hi]); return i+1。
4. マージは順次アクセスだけで済み、ランダムアクセス不要。ポインタの付け替えで追加メモリも抑えられる。
5. 大局は O(n log n) のマージで安定性を確保し、小区間は定数倍が軽い挿入ソートが速いから。実データに多い「既に並んだ連続部分(ラン)」を活かすため。
</details>
