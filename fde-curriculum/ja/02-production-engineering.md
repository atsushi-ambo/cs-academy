# モジュール 2 — プロダクションエンジニアリングの基礎（第4〜7週）

> ゴール: 実際のデータベースを備えたフルスタックアプリケーションを、コンテナ上、クラウド上で、
> 一人で設計・構築・デプロイ・デバッグできるようになること。これはあらゆるFDEの求人における最低ラインである。

FDEのコーディング面接は**実践的**であり、LeetCode偏重ではない。動くものを作り、APIを統合し、
壊れたサービスをデバッグし、データパイプラインを設計する。このモジュールは、週ごとに層を重ねていく
一つの連続したプロジェクトとして構成されている。

**通しプロジェクト「TicketFlow」** — 後に（モジュール5で）AIを後付けで組み込むことになる、小さな
社内ツールアプリ（サポートチケットのトラッカー）。社内ツールは、実際のFDE業務における日常の中核である。

---

## 第4週 — バックエンドとAPI設計

- [ ] **FastAPI**（Python）でREST APIを構築する — チケットのCRUD、ユーザー、コメント。
- [ ] 適切な構造化: ルーター、サービス、モデル。すべての境界でPydanticによるバリデーション。
- [ ] 認証: セッションまたはJWTログイン。JWTが*なぜ*ステートレスなのか、その代償が何かを理解する。
- [ ] ページネーション、フィルタリング、エラーエンベロープ、リクエストログのミドルウェア。
- [ ] テスト用データベースに対する`pytest`の統合テストを書く。

**学習**
- [ ] [FastAPI ドキュメント](https://fastapi.tiangolo.com/) — 「Tutorial – User Guide」を一通り読む。
- [ ] REST設計: [Zalandoのガイドライン](https://opensource.zalando.com/restful-api-guidelines/)を読む（流し読みでよい、リファレンスとして手元に置く）。
- [ ] HTTPを深く理解する: 冪等性、キャッシュヘッダー、コンテントネゴシエーション、ストリーミングレスポンス
      （モジュール5以降では絶えずトークンをストリーミングすることになる）。

## 第5週 — データベース（OpenAIの求人ではPostgresが名指しされている。そのヒントに乗ろう）

- [ ] TicketFlowを**Postgres**でモデリングする: スキーマ設計、外部キー、インデックス、制約。
- [ ] 最も有用なクエリ10本を手で書く（結合、集計、「エージェントごと・週ごとのチケット数」のための
      ウィンドウ関数）。この演習ではORMを使わない。その後にSQLAlchemyを追加して比較する。
- [ ] 遅いクエリに`EXPLAIN ANALYZE`をかけ、インデックスを追加し、その差を計測する。
- [ ] Alembicでマイグレーション。ダウンタイムなしのカラム名変更を練習する（expand → migrate →
      contract）。
- [ ] いつ次のものに手を伸ばすべきかを把握する: Redis（キャッシュ／キュー）、SQLite（組み込み／エッジ）、
      ドキュメントDB、ベクトルDB（モジュール5の予告）、データウェアハウス（BigQuery／Snowflake — 
      エンタープライズはこの中で生きている）。

**学習**
- [ ] [Use The Index, Luke](https://use-the-index-luke.com/) — 第1〜3章。
- [ ] トランザクションと分離レベル — Postgresドキュメントの該当章を読む。ロストアップデートとは何か、
      そして`SELECT ... FOR UPDATE`がどうそれを防ぐのかを説明できるようにする。

## 第6週 — フロントエンド、統合のための接着剤、データパイプライン

FDEは絶えずデモや社内UIを出荷する。「素早く、十分に良い」が基準である。

- [ ] TicketFlowのUIを**React + TypeScript**（Vite）で構築する: 一覧、詳細、作成、コメント。
- [ ] ストリーミングに適したデータ取得（トークンストリーミングで再利用する）: `fetch` +
      `ReadableStream`またはSSE。
- [ ] コンポーネントキットを一つ習得する（shadcn/uiまたはMUI） — 独創性よりもスピードが重要。

**統合のための接着剤 — FDE業務の隠れた半分。** エンタープライズの問題のほとんどは「システムAが
システムBと通信しなければならない」というものである。
- [ ] 雑なCSVエクスポート（自分で作る: 5万行、壊れたエンコーディング、重複キー、不整合な日付）を
      Postgresに取り込むパイプラインを書く: バリデーション → 正規化 → upsert → 拒否レポート。
      冪等で再実行可能にする。
- [ ] Webhookを受け取り（例: GitHub webhook → チケット自動作成）、リトライ + 指数バックオフ + 
      タイムアウト予算付きでサードパーティAPIを呼び出す。

## 第7週 — コンテナ、クラウド、CI/CD、可観測性

- [ ] バックエンドとフロントエンドをDocker化する。`docker compose up`でPostgresを含むスタック全体が起動する。
- [ ] 自分のクラウドアカウントにデプロイする: マネージドな経路を一つ（Cloud Run／App Runner／Azure Container
      Apps）**かつ**VPCの話を理解する（モジュール7で深掘りする）: サブネット、セキュリティ
      グループ、プライベート対パブリックのエンドポイントとは何か。
- [ ] GitHub Actions: PRでlint + テスト、mainへのマージでビルド + デプロイ。
- [ ] 可観測性の最低ライン: 構造化されたJSONログ、`/healthz`エンドポイント、エラートラッキング（Sentry
      無料枠）、そしてダッシュボード一つ（リクエストレート、p95レイテンシ、エラーレート）。
- [ ] プレッシャー下でのデバッグを練習する: 友人（またはスクリプト経由で自分自身）にデプロイ済みの
      アプリを3通りに壊してもらう — 不正な環境変数、DB接続の枯渇、依存サービスの障害 — 
      それぞれをログだけから修正する。時間を計る。これはFDEのオンコールの現実をシミュレートする。

**学習**
- [ ] [The Twelve-Factor App](https://12factor.net/) — 全部読む。短い。
- [ ] Designing Data-Intensive Applications（Kleppmann）、第1〜3章 — 読む。第5、7章 — 流し読み。
      その語彙（レプリケーション、パーティショニング、トランザクション）はFDEのシステム設計で登場する。
