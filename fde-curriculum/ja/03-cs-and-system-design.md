# モジュール 3 — CSの基礎とシステム設計（第8〜10週）

> ゴール: 「デモは作れる」と「Fortune 500企業の内部で一人で信頼を置ける」を分ける、コンピュータ
> サイエンスの深さとアーキテクチャの判断力。OpenAIのFDE採用ループにはコーディングラウンドと
> システム設計ラウンドが含まれる。PalantirのFDSEループは、分解と設計に強く依拠している。
> このモジュールは、その両方で危険なほど有能な人材を作り上げ、後続のすべてのモジュールに語彙を与える。

これは*FDE向けに調整された*分量である。研究職トラックの準備のような9か月の苦行なしに、ラボの
面接を突破し、本番システムについて推論できるだけの深さを備えている。任意のトピックについて
本格的な深掘りを望むなら、親リポジトリ — [Coding Interview University](https://github.com/jwasham/coding-interview-university) — が
標準的な地図である。これはこのモジュールの任意の拡張パックとして扱おう。

## 第8週 — データ構造とアルゴリズム（実用の作業集合）

FDEのコーディングラウンドは*実践寄り*だが、それでも流暢さは前提とされる。実際に動くもの（レート
リミッター、ログパーサー、ミニKey-Valueストア）を作るよう求められ、あなたのデータ構造の選択が
評価対象となる。LeetCodeのハード級の動的計画法はまれである。時間的プレッシャー下での、クリーンで
速く正確な実装は普遍的に求められる。

### 中核となる流暢さ（記憶から実装でき、選択できること）

- [ ] **配列、文字列、ハッシュマップ／セット** — 実践ラウンドの80%はここに存在する。衝突の
      振る舞い、O(1)償却計算量の注意点、Python／JSにおける辞書の順序保証を理解する。
- [ ] **スタック、キュー、デック** — パース、undo、BFSのフロンティア、スライディングウィンドウ。
- [ ] **ヒープ／優先度付きキュー** — top-K、スケジューリング、merge-K-streams（ページネーション
      されたAPI結果のマージで文字通りこれを使うことになる）。
- [ ] **木** — 走査（BFS／DFS、再帰的・反復的）、概念としてのBST。接頭辞問題のためのトライ木。
- [ ] **グラフ** — 隣接リスト、BFS／DFS、トポロジカルソート（依存関係の解決 — データパイプライン
      の仕事で絶えず現れる）、最短経路（実用レベルでBFS／Dijkstra）。
- [ ] **ソートと探索** — 組み込みソート + カスタムキー。二分探索とそのoff-by-oneの罠。実際の
      データサイズでO(n log n)とO(n)のどちらが効いてくるか。
- [ ] **計算量** — 時間*と*空間のBig-Oの流暢さ、加えて実践的な感覚の層:
      「n = 5万行 → O(n²)は何であれ死ぬ。n = 200 → 誰が気にする」。

### 練習プロトコル（1日3〜4問、今週は合計約25問、その後は維持）

- [ ] 厳選リスト: [NeetCode 150](https://neetcode.io/) — Arrays/Hashing、Two Pointers、
      Stack、Binary Search、Trees、Graphs、Heapのセクションをやる。今は風変わりなDPは飛ばす。
- [ ] **実践ドリル**（実際のFDEラウンドに近い — 45分のタイマー付きですべてやる）:
      - TTL有効期限付きのインメモリKey-Valueストアを実装する。その後トランザクションを追加する。
      - スライディングウィンドウ方式のレートリミッターを実装する。その後APIキーごとに対応させる。
      - 1GBのログファイル（生成する）をパースし、エンドポイントごとのp95レイテンシを計算する — 
        全部をメモリに読み込まずに。
      - LRUキャッシュを実装する（定番であり、いつか本当にデプロイすることになる）。
      - 深くネストされたJSON設定をドット区切りのキーにフラット化し、また元に戻す。
      - 「同一の」テーブルの2つのCSVエクスポートをdiffし、insert／update／deleteを出力する（これは
        面接問題に偽装された、実際に繰り返し発生するFDEのタスクである）。
- [ ] **声に出して**練習する: コードを書きながらトレードオフを語る。FDEの面接官はコーディング
      しながらのコミュニケーションを重く採点する — それは顧客先での振る舞いを予測するからである。

## 第9週 — マシンが実際にどう動くか（OS、ネットワーク、並行性）

あなたは*他人の*環境で本番システムをデバッグする。この層こそが、何も動かず誰も理由を知らない
ときに、あなたを速くするものである。

### オペレーティングシステムとLinux（FDEのサバイバルキット）

- [ ] プロセス対スレッド。メモリレイアウト（スタック／ヒープ）、仮想メモリ、OOM killとは何か、
      それをどう見分けるか（`dmesg`、終了コード137 — 顧客のk8s podでこれに出会うことになる）。
- [ ] ファイルディスクリプタ、パイプ、シグナル、終了コード。なぜ「too many open files」が起きるのか。
- [ ] Linuxの流暢さドリル — クラウドVM上で、GUIなしで: CPUを食っているプロセスを見つけ
      （`top`／`htop`）、その開いている接続を調べ（`ss`／`lsof`）、ログをtailしてフィルタリングし、
      ディスクを確認し（`df`／`du`）、ホストからコンテナを調べ（`docker exec/stats/logs`）、
      systemdユニットを読む。時間制限付きシナリオ: サービスがダウンしている — SSHだけで診断する。
- [ ] 読む: [The Linux Command Line](https://linuxcommand.org/tlcl.php)（パート1〜3をリファレンス
      として流し読み）、または[OverTheWire Bandit](https://overthewire.org/wargames/bandit/)の
      レベル1〜20をやり込む（より楽しく、同じスキルが身につく）。

### ネットワーク（あらゆるエンタープライズの問題は、密かにネットワークの問題である）

- [ ] リクエストの経路を、暗記で: DNS → TCPハンドシェイク → TLS → HTTP。それを語り、各ホップを
      デバッグできるようにする（`dig`、`curl -v`、`openssl s_client`）。
- [ ] HTTP/1.1対HTTP/2。keep-alive。ロードバランサーが何をするか（L4対L7）。リバースプロキシ。
- [ ] 長寿命の接続: WebSockets対SSE対ポーリング — そしてなぜトークンストリーミング
      （モジュール4以降）がほぼ常にSSEに乗るのか。
- [ ] エンタープライズネットワークの現実: プロキシ、egressファイアウォール、プライベートDNS、VPN、
      mTLS。なぜ「自分のラップトップでは動くのに、顧客のVPCでは403」が起きるのか — もっともらしい
      原因を5つ挙げる。

### 並行性（LLMアプリが実際にボトルネックになる場所）

- [ ] スレッド対プロセス対async — そしてPythonのGIL: それぞれが何のためにあるのか。あなたが
      擁護できる経験則: I/Oバウンドのfan-out（API呼び出し!）にはasync、CPUバウンドにはプロセス、
      ブロッキングI/Oライブラリにはスレッド。
- [ ] 作る: 1,000個のURLをフェッチ（または1,000回のモックLLM呼び出し）を、逐次で、スレッド
      プールで、`asyncio` + セマフォで上限を設けた並行性で。3つすべてを計測する。async版にリトライ + 
      タイムアウト + レートリミット処理を追加する — このまさに同じハーネスが、モジュール6の
      信頼性の作業で再び現れる。
- [ ] 競合状態とロックを実用レベルで。分散システムの非常口としての冪等性（モジュール2の
      パイプラインで出会った。今はそれに名前を与える）。

## 第10週 — システム設計とアーキテクチャ

FDEのシステム設計ラウンドは、標準的なSWEのものとは異なる。「5億ユーザー向けのTwitterを設計せよ」
より、「*この顧客*のための実際のシステムを、*彼らの*制約の下で設計し、技術／非技術が混在する
聴衆に対してトレードオフを擁護する」に近い。

### 構成要素（それぞれが何のためにあるか、その障害モード、実際の製品名を一つ知る）

- [ ] ロードバランサー。水平対垂直スケーリング。ステートレスなサービス（12-factorが報われる）。
- [ ] キャッシュ層（クライアント、CDN、Redis、DB） — そして無効化戦略を、正直に。
- [ ] **キューと非同期ワーカー**（SQS／RabbitMQ／Celery）: LLMシステムで最もよく使われる単一の
      パターン — 「ユーザーがクリックした」を「40秒後にモデルが完了した」から分離する。
- [ ] データベースのスケーリング: リードレプリカ、コネクションプーリング、シャーディング（そして
      なぜそれを遅らせるか）。いつ検索インデックス（Elasticsearch）やウェアハウス
      （BigQuery／Snowflake）を追加するか。
- [ ] レプリケーションと整合性: leader/follower、結果整合性、会話レベルのCAP。exactly-once対
      at-least-onceの配信（そしてなぜ「at-least-once + 冪等なコンシューマー」が現実世界の答えなのか）。
- [ ] バッチ対ストリーミングのデータ処理。ETL／ELT。Spark／Kafkaが何のためにあるか（Palantir風味の
      FDEループはこの語彙を期待する）。
- [ ] アーキテクチャパターンと、それぞれがいつ正しいか: モジュラーモノリス（デフォルト!）、
      マイクロサービス、イベント駆動、サーバーレス。5人のチームに対してマイクロサービスに
      *反対する*論を述べる準備をしておく。

### 手法（自動的になるまで反復する）

1. 要件 — 機能 + スケール + 制約（*設計の前に質問する。FDEのボーナス: 顧客の既存スタックと
   セキュリティ姿勢について尋ねる*）。
2. ナプキン計算 — QPS、ストレージ、帯域幅、コスト。練習: トークン/日 → $/月がFDE特有の捻りで
   ある。モジュール4でスプレッドシートをやった — 今度はホワイトボードでやる。
3. APIとデータモデルのスケッチ。
4. 高レベルのボックスと矢印。デフォルトでは退屈な技術を選ぶ。
5. 面接官が舵を切るところを深掘りする。聞かれる*前に*障害モードを名指しする。
6. トレードオフを明示的に述べる。面接の途中で要件が変わったら設計を進化させる（変わる — 顧客が
   スコープを変えるのにどう対処するかのテストである）。

### 練習設計（実際のホワイトボード／紙で4つ以上、声に出して。各35分）

- [ ] 1,000万ドキュメント、ユーザーごとの権限を持つドキュメントQ&Aシステム（モジュール5/7の
      予告 — 今は素朴に設計し、モジュール7の後に再設計して自分と差分を取る）。
- [ ] 銀行向けのLLM駆動のメール振り分けシステム: 100万メール/日、監査要件、p95で4秒。キューは
      どこに置くか? 人間はどこに置くか?
- [ ] 顧客のSalesforce + Postgres + S3のCSVを、データ品質チェック付きで一つのウェアハウスに
      同期する夜間パイプライン。
- [ ] ストリーミング、会話履歴、5万の同時接続ユーザーを持つチャットアプリのバックエンド
      （そう、これは「ChatGPTライトの設計」である — ラボは自社製品について尋ねる）。
- [ ] バースト的なトラフィックと不安定なダウンストリームを伴う、毎秒1万イベントのWebhook取り込み。
