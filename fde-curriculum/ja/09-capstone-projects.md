# モジュール9 — キャップストーン: 模擬顧客エンゲージメント（第26〜28週）

> 目標: **FDEループ全体** — ディスカバリー → スコープ → 構築 → 評価 → ハードニング → 引き継ぎ —
> を、時間的プレッシャーのもとで、エンドツーエンドで3回実行する。これらは面接でデモするプロジェクト
> であり、行動面接（behavioral round）で語るストーリーになる。それぞれを、実在する（模擬の）顧客との
> 本物のエンゲージメントとして扱おう。

**エンゲージメントのルール**
1. 各キャップストーンには**1週間という厳格なタイムボックス**が課される。小さなスコープを期限内に
   出荷する方が、大きなスコープを遅れて出荷するより優れている — それ*こそ*が学びだ。
2. 各キャップストーンには「顧客」が必要だ: ペルソナを演じる友人、または詳細なペルソナプロンプトを
   持ったLLMで、要件を聞き出すために面談し、毎週デモする相手。これを省略してはならない。これらの
   会話から生まれる成果物が、ポートフォリオの価値の半分を占める。
3. すべてのキャップストーンは完全な成果物一式を生み出す: 簡易版SOW → 動作するデプロイ済みシステム →
   数字を伴う eval レポート → セキュリティノート → エグゼクティブ向けワンページャー → ランブック →
   5分のデモ動画。
4. 各キャップストーンの主要モデルプロバイダーを使い分ける（Claude／GPT／Gemini）。少なくとも1つは
   クラウド経由のルート（Bedrock／Vertex／Azure）を通すこと。

## キャップストーンA（第26週）— 規制業界向けのドキュメントインテリジェンス

**ペルソナ**: 中堅保険会社のオペレーション責任者。年間3万件の保険金請求がPDF（スキャン、写真、
フォーム）で届く。査定担当者はデータを再入力し、約款の免責事項を見落とす。失敗したチャットボット
パイロットの後でAIに懐疑的。セキュリティチームは厳格。

**想定される形**（ディスカバリーで決めるが、こう予想される）: 雑然としたPDFからのマルチモーダル抽出 →
信頼度スコア付きのスキーマ検証済みフィールド → 約款文書に対するRAGによる免責事項のフラグ付け →
信頼度の低い項目を人にルーティングする人手レビューキュー → 指標ダッシュボード（ストレートスルー率、
削減分数）。あなたが組み立てる、ラベル付けされた50件以上の文書からなる eval スイート（公開されている
保険フォーム、合成された請求データ）。

**何を証明するか**: マルチモーダル処理、抽出の eval、ヒューマン・イン・ザ・ループ設計、規制業界の
セキュリティ態勢（モジュール7のブリーフィングを適応させたもの）。

## キャップストーンB（第27週）— ビジネスシステム内部でのエージェント自動化

**ペルソナ**: B2B SaaSスタートアップのカスタマーサクセス責任者。サポートはメール＋共有受信箱＋
wiki＋Stripeで運用されている。「AIにサポートを処理させたい」（あなたはこれをスコープダウンしなければ
ならない）。誤った返金やハルシネーションによる約束を恐れている。

**想定される形**: トリアージ兼ドラフト作成エージェント — 受信したリクエストを分類し、ツール経由で
アカウントのコンテキストを取得し（モックまたはサンドボックスのStripe＋あなたが作成するwiki）、引用付き
で回答を下書きし、証明可能に安全なカテゴリのみを自動解決し、それ以外はすべて事前入力済みのサマリーと
ともに人にルーティングする。ビジネスシステムを公開するMCPサーバー、承認キュー、トラジェクトリ eval
（ツール呼び出しの正確さ、エスカレーションの正しさ）、チケットあたりの支出／レイテンシ予算。

**何を証明するか**: エージェント＋MCP＋ツールの権限管理（tool permissioning）＋過度に広い要求を
スコープ*ダウン*する判断力 — そのスコーピングの判断をデモ動画で語ること。それがポートフォリオの中で
最もシニアらしさが出る瞬間だ。

## キャップストーンC（第28週）— エンタープライズデータに対するアナリティクスコパイロット

**ペルソナ**: 小売チェーンのCFO。アナリストは毎月、「なぜ地域Xで利益率が下がったのか？」に答えるのに
何日も費やす。データはウェアハウスにある（Postgres／DuckDB＋公開小売データセットでシミュレートする）。
間違った数字は、数字がないことより悪い。

**想定される形**: スキーマグラウンディングを伴うtext-to-SQLと、読み取り専用・許可リスト方式の接続 →
結果の検証（行数、サニティチェック、「SQLを見せて」という透明性）→ チャート／サマリーの生成 →
実行ベースの eval スイート（50組の質問／SQL／回答 — テキストを判定するのではなく、クエリを実行して
採点する）→「分からない／曖昧な質問」という拒否挙動、テスト済み。

**何を証明するか**: データエンジニアリング＋責任を持って行われたtext-to-SQL＋実行ベースの eval ＋
取締役会であなたの数字を引用するエグゼクティブに対する不確実性の伝え方。

## ポートフォリオの組み立て（継続的に進め、第28週末に完成させる）

- [ ] ポートフォリオのREADMEを、プロフェッショナルなインデックスとして書き直す: プロジェクトごとに
      1段落＋指標＋デモリンク。TicketFlow AIと3つのキャップストーンを先頭に置く。
- [ ] **公開ブログ記事を1本書く**。ぶつかった中で最も興味深い技術的問題について（例:「エージェントを
      返金処理の前に置いたとき、何が壊れたか」）。各ラボのリクルーターや採用マネージャーは本当にこれを
      読んでいる。良い記事1本は、リポジトリ10個に勝る。
- [ ] すべてを相互リンクする: 各リポジトリのREADME → ライティング成果物 → デモ動画 → ブログ記事。
- [ ] 正直な失敗ページ: プログラム全体で*うまくいかなかった*ことと、それを受けて何を変えたかを列挙
      するセクションを1つ。逆説的だが、これこそ面接官が記憶するページだ。
