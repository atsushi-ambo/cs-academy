# モジュール8 — 顧客スキル: ディスカバリー、スコーピング、エグゼクティブとのコミュニケーション（第24〜25週）

> 目標: エンジニアが体系的に過小評価しがちなFDE業務のもう半分を身につける。Anthropicの
> 採用ループには、顧客との会話を模擬したシミュレーションがあり、報じられるところでは
> *すでにコーディング選考を通過した*候補者のうち約60%をふるい落とすという。これは前週に
> 一夜漬けで詰め込めるものではない。集中的な2週間で訓練し、その後も練習を続けられるものだ。

ここで扱うすべては**声に出し、また文章として書いて**練習する — コミュニケーションについて
読むだけでは何も鍛えられない。

## 第24週 — ディスカバリーとスコーピング

### レッスン8.0 — ビジネスの基礎: 企業はどのように購入し、「価値」とは何を意味するのか

価格をつけられないものはスコープできない。会話スキルに入る前に、あなたの仕事が存在する基盤と
なるビジネスの仕組みを自分のものにしておこう:

- [ ] **エンタープライズソフトウェアがどのように購入されるか**: チャンピオン → 経済的バイヤー
      （economic buyer）→ セキュリティレビュー → 調達（procurement）→ 法務。FDEが各段階で誰に
      仕えるのか、そしてなぜ各段階で商談が頓挫するのか（価値が定量化されていない、セキュリティ
      レビューで不合格、あなたが去った後に社内のオーナーがいない）を理解する。
- [ ] **これから一生使い回すROIの方程式**:
      `価値 = (削減時間 × 人件費込みの時間単価) + 売上増 + リスク低減 − (構築コスト
      + 運用コスト + 保守コスト)`。TicketFlow AIで練習しよう: 1日あたりのチケット数 × 削減分数 ×
      エージェントコスト を、モジュール6で算出したチケット単価の数値と比較する。結果をエグゼク
      ティブ向けワンページャー（レッスン8.3）に記載しよう —「年間31万ドルを節約し、運用に年間
      1万8千ドルかかる」というデプロイメントこそが、FDEの語り方だ。
- [ ] **パイロット → 本番（production）の経済性**: なぜパイロットは1つのチームと明確な指標に
      絞ってスコープするのか。「land and expand（着地して拡大する）」とは何を意味するのか。
      なぜ計測されていない成功したパイロットは失敗したパイロットなのか。
- [ ] **構築 vs 購入 vs 待機（build vs buy vs wait）**: 与えられたユースケースについて、3つすべて
      を誠実に論じられるようにする —「2世代分のモデルを待つ」という選択肢も含めて。これは時に
      正しい判断であり、常に信頼を築く材料になる。
- [ ] **2時間で顧客のビジネスを読み解く**: あなたが興味のある業界の公開企業を1社選び、最新の
      年次報告書／決算説明会のサマリーにざっと目を通し、次の問いに答える: この会社はどうやって
      利益を上げているか、最大のコストセンターは何か、AIが彼らの報告する数字をどこで現実的に
      動かしうるか。10行書く。これを2回やる。（FDEは初回ミーティングの前にまさにこれをやる。）

### レッスン8.1 — ディスカバリー: 本当の問題を見つける

顧客は解決策を語る（「チャットボットが欲しい」）のであって、問題を語る（「うちの担当者は
時間の40%を古いチケットの検索に費やしている」）のではない。ディスカバリーとは、前者から後者へ
たどり着くための構造化された会話だ。

- [ ] 質問の階段（question ladder）を学び、自分なりのバージョンを書く:
      - *ワークフロー*:「これが最後に起きたときのことを、ステップごとに順を追って教えてください。」
      - *痛みの定量化*:「どのくらいの頻度で？どれくらい時間がかかる？誰がやる？エラーが起きると
        どれだけのコストになる？」
      - *現状の回避策*:「これまで何を試しましたか？なぜ定着しなかったのですか？」
      - *成功の定義*:「これが完璧に機能したら、誰のダッシュボード上のどの数字が変わりますか？」
      - *制約*: データアクセス、セキュリティ、関係するシステム、誰の承認が必要か。
- [ ] 自分から叩き出すべきアンチパターン: 理解する前に売り込む。提示された解決策を鵜呑みにする。
      1人のステークホルダーとしか話さない。ディスカバリー中に専門用語を使う。「AIがXをエンドツー
      エンドでやるべきだ」を聞いて、80%の価値をもたらす20%のスライスを探らない。
- [ ] **練習（必須、人間を相手に）**: 友人や同僚を募り、1段落のペルソナを渡す（「あなたは保険会社で
      保険金請求処理を統括している。あなたは懐疑的だ。ベンダーにまつわるひどい体験談を持っている」）。
      30分のディスカバリーコールを実施する。録音する。聞き返して — 何回相手をさえぎったか、早まって
      売り込んだか、誘導的な質問をしたかを数える。2つ目のペルソナでもう一度やる。（本当に人間が
      手配できない場合は、顧客を演じるLLMを相手に実施する — 人間より劣るが、何もしないよりはるかに
      ましだ。）

**学習**: *The Mom Test*（Rob Fitzpatrick）— 短く、顧客への質問の仕方について書かれた唯一最高の本。
創業者向けに書かれているが、FDEのディスカバリーに1対1で対応する。

### レッスン8.2 — スコーピング: 問題からスティールスレッドへ

- [ ] **スティールスレッド（steel thread）**: 中核的な価値を証明する、最も細いエンドツーエンドの
      スライス（すべてのカテゴリを半分だけ自動化するのではなく、1つのチケットカテゴリを完全に
      自動化する）。練習: 5つの問題ステートメント（自分の知る業界から考案する）について、それぞれ
      スティールスレッドのスコープを5文で書く。
- [ ] **実現可能性トリアージ（feasibility triage）** — FDEの判断の見せどころ。提案されたあらゆる
      AIユースケースについて評価する: データはアクセス可能で十分に良質か？タスクは検証可能か
      （eval を構築できるか）？間違った答えのコストはどれくらいで、人間のチェックポイントはあるか？
      今日のモデル能力で実際にカバーできるか？10のユースケースで練習し、*今すぐ構築する／慎重に
      パイロットする／辞退する*に分類する。「辞退する、なぜなら…」と言えることはシニアであることの
      シグナルだ。
- [ ] TicketFlow AIについて、顧客エンゲージメントのつもりで**簡易版SOW（Statement of Work-lite）を
      書く**: 問題、スコープ（含むものと含まないもの — この「含まないもの」のリストこそプロジェクトが
      頓挫する場所だ）、デモ日を含むマイルストーン、成功指標、緩和策を伴うリスク、顧客から必要な
      もの（データアクセス、週2時間のドメインエキスパート、セキュリティレビューの枠）。テンプレート化し、
      一生使い回す。

## 第25週 — コミュニケーションの成果物とエグゼクティブ・プレゼンス

### レッスン8.3 — ライティング（FDEは絶え間なく書く）

TicketFlow AIについて、4つの成果物すべてを本気で `writeups/` に作成する:

- [ ] **デザインメモ**（1〜2ページ）: 背景 → 検討した選択肢 → 推奨案 → トレードオフ → 必要な決定。
      トップダウン構造: 忙しい読者は最初の5行ですべてを把握できる。
- [ ] **週次ステータスアップデート**（最大10行）: 出荷したもの／指標の動き／ブロックされている点／
      来週の予定／リスク。何かが遅延した想定上の第3週のアップデートを書く — 悪い知らせを早く、率直に
      報告する練習をする。
- [ ] **エグゼクティブ向けワンページャー**: 何を構築したか、それが動かした数字、運用コスト、次の一手。
      専門用語ゼロ。テスト: 技術系でない友人が3分で読み、その要点を言い返せること。
- [ ] **引き継ぎ／ランブック文書**: あなたが去った後、システムをどう運用し、監視し、拡張するか。
      （FDEは去る。この文書こそ、拡大を可能にする成果物だ。）

### レッスン8.4 — デモとエグゼクティブとの会話

- [ ] **デモの技術**: 問題 → ライブの製品 → 数字 → 次の一手、と語る。機能リストからデモしては
      ならない。ユーザーの1日からデモする。常に録画したバックアップを用意する。意図的に処理済みの
      失敗を1つ見せる。モジュール5のデモを、これらすべてを適用して録り直し、比較する。
- [ ] **翻訳ドリル**: 以下のそれぞれを、賢い非エンジニアに、声に出して、各60秒以内で、録音しながら
      説明する — コンテキストウィンドウ。なぜモデルは「嘘をつく」のか（ハルシネーション）。なぜ出荷前に
      eval スイートが必要なのか。なぜこれはチケットあたり0.04ドルかかるのか。プロンプトインジェクションとは何か。
      聞き返す。専門用語を排除する。クリーンになるまで繰り返す。
- [ ] **押し戻しシミュレーション（pushback simulation）**（面接ラウンドのリハーサル）: 友人かLLMに
      演じてもらう — 不可能な締め切りを要求するエグゼクティブ。モジュール7の反論を持つCISO。脅威を
      感じてひそかにあなたを妨害する顧客側のエンジニア。それぞれのゴール: 技術的な一線を守り*かつ*
      関係を維持すること。打ち手は常に: 認める → 共通のゴールに再フレーミングする → 理由とともに
      実現可能な代替案を提示する → テストについて合意する。
- [ ] **ステークホルダーマッピング**: あなたの模擬エンゲージメントについて、スポンサー／ユーザー／
      妨害者（blockers）／セキュリティ／ITを図にする。それぞれが何を気にかけているか、それぞれが毎週
      あなたに何を必要としているか。
