# モジュール6 — 評価、Observability、信頼性（第19〜21週）

> 目標: 「どうやってそれが機能していると分かるのか？」に — 数字で — 答えられる、その場の唯一の
> 人物になること。evalはFDE/Applied-AIの面接ループで最も頻繁に挙げられる差別化要因であり
> （Anthropicのシステム設計ラウンドは明確にevalに焦点を当てている）、候補者に最もよく見られる
> 弱点でもある。

このモジュールを定義する、顧客との会話:
> **顧客:** 「いくつかの例で試したら、良さそうでした。出荷していいですか？」
> **あなた:** 「こちらがevalスイート、スコア、失敗の分類体系、そしてリグレッションゲートです。
> これで出荷できます。」

---

## 第19週 — evalの設計

- [ ] **「雰囲気」から「指標」へ**: 曖昧な目標（「回答は良くあるべき」）を、測定可能な基準
      （根拠性、網羅性、フォーマット遵守、トーン、拒否の正しさ）に変換する。
- [ ] **ゴールデンデータセットを構築する** — TicketFlow AI向けに100以上のラベル付きの例。出所:
      合成生成（モデル支援、人間が検証）、Field Notesからのエッジケース、敵対的ケース。なぜ
      30の良い例が1000の雑な例に勝るのかを学ぶ — そして、なぜエンゲージメントの初日から収集を
      始めるのかを学ぶ。
- [ ] **grader（採点器）の種類** と、それぞれが信頼できるのはいつか:
      - 完全一致 / 正規表現 / スキーマチェック（安価、脆い、抽出とSQLには最適）、
      - 実行ベース（生成されたSQL/コードを実行し、結果を比較する — 可能なときは最も強力）、
      - **LLM-as-judge**（ルーブリック設計、位置バイアス、自己選好バイアス。信頼する前に約30件の
        人間ラベルに対してjudgeをキャリブレーションする）、
      - 人間によるレビュー（あなたが節約しようとしている、ゴールドスタンダード）。
- [ ] evalハーネスを実装する: （データセット、システムのバージョン）を受け取り → スコアカードを
      返す、`pytest` スタイルのランナー。RAGパイプラインの3つのプロンプトのバリアントに対して
      実行し、**データで** 勝者を選ぶ。
- [ ] **agentのeval**（より難しく、面接で好まれる）: 最終的な回答だけでなく、軌跡（trajectory）を
      評価する — 正しいツールを、正しい引数で、まともな順序で呼んだか？ ツール呼び出しの精度、
      タスク完了率、完了までのステップ数、タスクあたりのコスト。
- [ ] RAG特有: 検索の指標（recall@k、MRR）を、生成品質とは **別々に** 測定する。これにより、
      どちらの半分が壊れているかが分かる。

**学習素材**
- [ ] Anthropicのドキュメント — [経験的evalガイド](https://docs.anthropic.com/en/docs/test-and-evaluate)。
- [ ] Hamel Husain — [「Your AI Product Needs Evals」](https://hamel.dev/blog/posts/evals/) と、その
      エラー分析に関する続編。じっくり読むこと。これは実務家の正典だ。
- [ ] evalフレームワーク（promptfoo / Braintrust / LangSmith evals）にざっと目を通し、自作の
      ハーネスに何が欠けているかを判断する。

## 第20週 — エラー分析と改善ループ

- [ ] **エラー分析の儀式**: スイートを実行する → すべての失敗を読む（そう、手作業で）→ 自分で考案した
      失敗の分類体系（検索の失敗 / 推論の失敗 / フォーマット / 拒否 / データのバグ）でタグ付けする →
      最大のカテゴリを修正する → 再実行する。TicketFlow AIに対して **3周の完全なサイクル** を回し、
      スコアをグラフ化する。このループ *こそが* 応用AIエンジニアリングである。面接官はこれを語る
      ことを求めてくる。
- [ ] **CIでのリグレッションゲート**: プロンプトに触れるすべてのPRでevalを実行する。スコアの低下は
      マージをブロックする。モジュール2のGitHub Actionsに組み込む。
- [ ] **ドリフト**: モデルのバージョンアップグレードは、挙動を静かに変える。バージョンをピン留め
      する、アップグレード時にスイートを再実行する、依存関係と同じようにモデルアップグレードの
      変更履歴を残す。
- [ ] **オンラインのシグナル**: 高評価/低評価、ドラフトと人間が実際に送った内容との編集距離、
      エスカレーション率。フィードバックの収集を製品のUIに設計として織り込む。

## 第21週 — LLMシステムのためのObservabilityと信頼性

- [ ] **トレーシング**: すべてのリクエストにトレースを付ける — プロンプトのバージョン、モデル、
      トークン、コスト、レイテンシ、ツール呼び出し、検索されたチャンク。TicketFlow AIを計装する
      （OpenTelemetry、またはLangSmith/Langfuse/Braintrustのようなプラットフォーム — 1つ選んで
      深く掘る）。
- [ ] **顧客の経営幹部が読めるダッシュボード**: 日あたりのコスト、p50/p95レイテンシ、成功率、
      エスカレーション率。これを構築する。
- [ ] **信頼性のパターン**: ステップごとのタイムアウト予算、ジッター付きのリトライ（冪等性！）、
      **フォールバックチェーン**（主モデル → より安価なモデル → 静的なレスポンス）、サーキット
      ブレーカー、スパイク負荷に対するキュー + 非同期、プロバイダーの障害時の優雅な劣化（障害は
      起きる — それを前提に設計し、各プロバイダーのステータスページの履歴を把握しておく）。
- [ ] **本番でのコスト管理**: キャッシュ（完全一致 + プロンプトキャッシュ）、モデルルーティング
      （簡単なケースには安価なモデル、難しいものはエスカレーション）、コンテキストの剪定、
      オフライン作業のためのバッチAPI。演習: evalスコアを落とさずに、TicketFlow AIのチケット
      あたりのコストを50%削減する。引いたレバーを文書化する — 面接で語る最高のストーリーになる。
- [ ] **インシデント訓練**: 「コードは変更していないのに、本番で品質が低下した」を再現する。診断を
      たどる: モデルがアップグレードされた？ コーパスが変わった？ トラフィックの構成が変化した？
      プロンプトキャッシュが汚染された？ ポストモーテムを `writeups/` に書く。

## モジュール6の成果物

`writeups/ticketflow-eval-report.md` — ゴールデンデータセットの説明、ハーネスの設計、3周の
改善グラフ、リグレッションゲートの設定、コスト削減の結果。この1つの文書を10分間声に出して
説明すること、それがあなたの面接における最強の武器だ。

## モジュール6の修了基準

- [ ] 面接官が挙げるどんなタスクに対しても、3分以内にevalを設計できる。網羅すべきは:
      データセット、grader、指標、そしてそれがどうデプロイをゲートするか。
- [ ] CIが品質を台無しにするプロンプト変更をブロックする — 意図的に悪いPRで実証する。
- [ ] LLM-as-judgeの落とし穴と、自分のものをどうキャリブレーションしたかを説明できる。
