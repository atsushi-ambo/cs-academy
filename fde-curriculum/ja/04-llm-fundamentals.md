# モジュール 4 — LLMの基礎とプロンプトエンジニアリング（第11〜13週）

> ゴール: LLMの振る舞いを、失敗が起きる前に予測できるほど十分に理解し、プロンプトでモデルを
> 確実に操縦すること。FDEはモデルを訓練しない — しかし*モデルの振る舞い*については、顧客の
> エンジニアたちよりも深く理解していなければならない。

---

## 第11週 — LLMが実際にどう動くか（FDEに関係する20%）

研究の深さではなく、機能する心的モデルが必要である。

- [ ] **トークン**: それが何か、なぜ「strawberryにrは何個」が失敗するのか、なぜトークン数が
      コストとレイテンシを左右するのか。直感が働くようになるまでトークナイザーの可視化ツールで遊ぶ。
- [ ] **次トークン予測とサンプリング**: temperature、top-p。なぜ出力が変動するのか。いつ
      temperatureを0に設定するか（そしてなぜそれでも完全には決定的でないのか）。
- [ ] **コンテキストウィンドウ**: 何が収まるか、何にコストがかかるか、「lost in the middle」の
      劣化、なぜ「とにかく全部コンテキストに詰め込む」がいずれ失敗するのか — そしていつそれが
      実際に問題ないのか。
- [ ] **訓練パイプラインの概観**: 事前学習 → 指示チューニング → RLHF／RLAIF。モデルがなぜ
      おべっか使い（sycophantic）なのか、なぜハルシネーションを起こすのか、システムプロンプトとは
      何かを説明できる程度に。
- [ ] **モデルの全体像**: 現行のClaude／GPT／Geminiファミリーとオープンウェイト（Llama、
      Mistral、Qwen）。それぞれについて: コンテキスト長、Mtokあたりのコスト、レイテンシのクラス、
      モダリティ。比較シートを作り、**毎月更新する** — 古いモデル知識は面接でひどく映る。
- [ ] **マルチモダリティ**: 画像／PDF／音声の入力。OCRでは解決できない問題をビジョンが解決するのはいつか。

**学習**
- [ ] Karpathy — [Intro to Large Language Models](https://www.youtube.com/watch?v=zjkBMFhNj_g)（1時間、最良の単一の土台）。
- [ ] Karpathy — [Deep Dive into LLMs like ChatGPT](https://www.youtube.com/watch?v=7xTGNNLPyMI)（任意だが優れている）。
- [ ] [3Blue1Brown — Transformers, visually](https://www.youtube.com/watch?v=wjZofJX0v4M)（直感のみ。数式は罪悪感なく飛ばしてよい）。

## 第12週 — プロンプトエンジニアリング、本気で

プロンプティングはFDEの主要な調整つまみである。即座に出荷でき、コストはゼロで、「モデルが
間違っている」という苦情の大半を解決する。

- [ ] **両方**のプロンプトエンジニアリングガイドを読む — [Anthropic](https://docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/overview)
      と[OpenAI](https://platform.openai.com/docs/guides/prompt-engineering)。両者が
      食い違うところに注目する。
- [ ] それぞれハンズオン演習で習得する: ロール／システムプロンプト。構造化出力（JSONモード／
      ツールスキーマで強制された出力）。few-shotの例（そしてそれが裏目に出るのはいつか）。chain of
      thoughtと「step by stepで考えて」対それをネイティブに行う推論モデル。長いコンテキストのための
      XML／markdownのセクション分け。指示を長いドキュメントの*後に*置くこと。プロンプト
      インジェクションへの意識（モジュール7で深掘り）。
- [ ] **推論モデル**（oシリーズ、Claudeの拡張思考、Geminiの思考）: 何のためにあるか、
      レイテンシ／$でいくらかかるか、顧客の問題がそれらを正当化するのはいつか。
- [ ] キャッシング: Anthropic & OpenAIでのプロンプト／コンテキストのキャッシング — 静的な90%が
      キャッシュされるようにプロンプトを再構成する。コストの差を計測する。

**演習 — プロンプト道場。** 5つのタスクを選ぶ（雑な請求書からフィールドを抽出する。サポート
チケットを分類体系に分類する。法律文書を中学2年生レベルに書き換える。質問 + スキーマからSQLを
生成する。ルーブリックに対してエッセイを採点する）。各々について: プロンプトを書き、20個のテスト
入力を作り、プロンプト**のみ**を反復することで精度90%以上を得る。すべての反復を記録する — 
「一つだけ変えて、計測する」という規律は、萌芽段階のモジュール6である。

## 第13週 — プロのようにモデルAPIを扱う

- [ ] Anthropic + OpenAI + Geminiの上に、薄い**プロバイダ非依存のクライアント**を構築する
      （または一つ学ぶ: LiteLLM／Vercel AI SDK）: ストリーミング、バックオフ付きリトライ、
      タイムアウト予算、トークンカウント、リクエストごとのコストロギング。
- [ ] **ストリーミングUX**: TicketFlowのフロントエンドにトークンをストリーミングする（SSE）。
      time-to-first-token対総レイテンシを計測して体に染み込ませる — 顧客はTTFTを感じ取る。
- [ ] **本番投入の構造化出力**: 無効時にリトライするスキーマ検証付き抽出。障害モードを知る
      （JSON途中での切り詰め、スキーマドリフト）。
- [ ] **コストとレイテンシのエンジニアリング**: 「サポートチケット5万件/日、入力2kトークン、
      出力300」のコストを3モデルにわたって見積もる。小さなスプレッドシートモデルを作る。FDE面接は
      ナプキン計算を好む。
- [ ] **レートリミットとクォータ**: キューを用いてRPM／TPM制限を尊重するワーカーを設計する。
- [ ] **ファインチューニング対プロンプティング対RAG**: それぞれが勝つのはいつか。デフォルトの
      答え: プロンプティング + RAG。スタイル／フォーマットを大規模に、あるいはレイテンシが
      重要な分類にはファインチューニング。この階層を擁護できるようにする — 標準的な面接の探りである。

**成果物 — 「モデル振る舞いフィールドノート」。** あなた自身が見つけた10個の驚くべきモデルの
振る舞いを文書化した公開リポジトリ／ページ（再現可能なプロンプト付き）: ハルシネーションの
パターン、フォーマットの癖、モデル間の不一致、コンテキスト位置の効果……。これは異例なほど強力な
ポートフォリオの一品である。なぜならFDEの中核スキル — *モデルの振る舞いに対する経験的な好奇心* — 
を示すからである。
