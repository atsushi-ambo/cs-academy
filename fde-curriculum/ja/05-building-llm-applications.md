# モジュール5 — LLMアプリケーションの構築: RAG、ツール、agent、MCP（第14〜18週）

> 目標: 応用AIにおける4つの中核となるパターンを習得する。AnthropicのFDE求人票には、成果物が
> 明確に記されている。*「本番ワークフローで使われるMCPサーバー、サブagent、agent skill」* である。
> このモジュールはカリキュラムの技術的な核心だ。

強く推奨するデフォルト: **プロバイダーのSDKを直接使い**、すべての構成要素を理解すること。
フレームワーク（LangChain/LlamaIndexなど）は、その後に高速化の手段として学ぶ。FDEの面接では、
フレームワークを操作することしかできない候補者は容赦なく落とされる。

---

## 第14週 — Retrieval-Augmented Generation（RAG）を、部品から組み立てる

モジュール0の「Ask My Docs」を、今度はきちんと作り直す。コーパスは実在する文書を約200点（例:
ある製品のドキュメント一式、またはWikipediaのサブセット）。

- [ ] **取り込み（Ingestion）**: パース（PDF/HTML/Markdown）、クリーニング、チャンク化。3つの
      チャンク化戦略（固定長、セマンティック/見出しベース、オーバーラップあり）を試す。どれが
      優れているかは、モジュール6できちんと測定する。
- [ ] **embeddingとベクトル検索**: ホスト型モデルでembeddingし、**pgvector** に保存する（あなたは
      すでにPostgresを動かしている。FDEの利点: 顧客のセキュリティチームに売り込むシステムが1つ減る）。
- [ ] **ハイブリッド検索**: BM25/全文検索 + ベクトル + **リランキング**（ホスト型リランカー）。
      ハイブリッドが純粋なベクトル検索を圧倒する場面は、恥ずかしくなるほど多い。その理由を理解する
      こと（厳密な識別子、専門用語、コード類）。
- [ ] **生成（Generation）**: チャンクへ遡って参照できる **引用（citation）** 付きの、根拠に
      基づいた回答。コーパスに答えが含まれない場合は回答を拒否する（流暢さよりも、企業にとっては
      こちらの方がずっと重要だ）。
- [ ] **失敗のツアー**: 意図的に次の事象を引き起こす — 悪いチャンクから自信満々に誤った回答を出す、
      「lost in the middle（中盤の見落とし）」による取りこぼし、ドキュメント更新後にインデックスが
      古いままで起きる取りこぼし。それぞれをField Notesに書き留める。
- [ ] スケーリングのストーリーを理解する: pgvectorが限界に達したら次は何か（専用のvector DB、
      マネージド検索）、メタデータフィルタリング、マルチテナントの分離。

**学習素材**: AnthropicとOpenAIのRAG/検索クックブック、[pgvectorのドキュメント](https://github.com/pgvector/pgvector)。

## 第15週 — tool use / function calling（agentへの入り口）

- [ ] **TicketFlowの実際のAPI** に対してモデルにツールを与える: `search_tickets`、`get_ticket`、
      `update_status`、`post_comment`。自然言語を入力 → 正しいAPI呼び出しを出力。
- [ ] ツール設計の技法: 説明文はプロンプトそのものである。数が少なく適切に命名されたツールは、
      数の多いツールに勝る。エラーはモデルが反応できる構造化されたコンテンツとして返す。ツール結果の
      ページネーション/切り詰め。
- [ ] 並列のツール呼び出し、複数ステップのチェーン（モデルがツールを呼ぶ → 結果を読む → 次を呼ぶ）。
- [ ] **Text-to-SQL** を責任を持って実装する: スキーマをコンテキストに入れる、読み取り専用接続、
      許可リスト、結果サイズの制限。TicketFlowのDBに対して構築する。（あらゆる企業がこれを求める。）
- [ ] 厄介な部分に対処する: 存在しないツールをモデルが呼ぶ、引数がバリデーションに失敗する、
      ツールがタイムアウトする。あなたのハーネスはこの3つすべてを生き延びなければならない。

## 第16週 — agent

agent = モデル + ツール + ループ + 停止条件。エンジニアリングは魔法ではなくループの中にある。

- [ ] 生のSDK呼び出しからTicketFlow向けの **サポートトリアージagent** を構築する: 新しい
      チケットを読み、ドキュメントを検索し（第14週のRAGをツールとして利用）、返信案を作成し、
      自分で処理できないものはエスカレーションする。最大反復数のガード、トークン/コストの予算、
      すべてのステップの完全なトレースログ。
- [ ] **Human-in-the-loop（人間の介在）**: agentは *提案* し、破壊的なアクションは人間が承認する。
      承認キューのUIを構築する。企業はこれなしにはデプロイしない。これを設計に織り込むFDEは
      取引を勝ち取る。
- [ ] **サブagent / オーケストレーション**: 専門化したワーカーに委譲するプランナー（トリアージ →
      {課金agent、技術agent}）。これが有効な場合（コンテキストの分離、並列性）と、単なる複雑さの
      コスプレに過ぎない場合の区別。
- [ ] **メモリと状態**: 会話の永続化、コンテキストが埋まったときの要約による圧縮、クラッシュ後の
      再開可能性（チャット履歴だけでなく、agentの状態機械そのものを永続化する）。
- [ ] 読む: [Anthropic — Building Effective Agents](https://www.anthropic.com/research/building-effective-agents)
      （ワークフロー vs agent、シンプルに始めること）。このエッセイの語彙は面接での共通言語だ。
- [ ] 本番のハーネスを1つ深く研究する（例: Claude Code、またはオープンソースのagentリポジトリ）:
      計画、ツールのパーミッション管理、コンテキストの圧縮をどう行っているか。

## 第17週 — MCP（Model Context Protocol）と統合レイヤー

MCPは「モデル ↔ 企業システム」をつなぐ標準的な配管であり、FDEの成果物として名指しされている。

- [ ] プロトコルを理解する: サーバー、クライアント、**tools / resources / prompts**、トランスポート
      （stdio、streamable HTTP）。[MCPのドキュメント](https://modelcontextprotocol.io/)を読む。
- [ ] **TicketFlow向けのMCPサーバーを構築する**（PythonまたはTSのSDK）: そのAPIをMCPツールとして
      公開し、Claude Desktop/Codeに接続して、チャットからアプリを操作する。
- [ ] もっと厄介なものをラップした2つ目のMCPサーバーを構築する — レガシーなSOAP/XML API、または
      CSVのディレクトリ。鍛えられるスキルは、*汚い企業システムを、モデルに向けたクリーンな
      インターフェースでラップすること* だ。この一文がFDEの仕事の半分である。
- [ ] 企業におけるMCPの認証: OAuthフロー、スコープを限定したトークン、ツール呼び出しの監査ログ。
- [ ] **agent skill**: 繰り返し可能な手順（例:「週次のチケット傾向レポート」）を、再利用可能な
      skill/プロンプト+スクリプトのバンドルとしてパッケージ化する。

## 第18週 — 統合と、フレームワークの概観

- [ ] **「TicketFlow AI」を出荷する**: RAGに基づいた回答 + トリアージagent + 承認キュー + MCP
      サーバーを、デプロイし、5分でデモできる状態にする。これはポートフォリオの目玉となる作品だ。
- [ ] **5分のデモ動画** を録画する: 課題、稼働する製品、1つの失敗ケースとシステムがそれをどう
      封じ込めるか。（意図的に失敗を見せるのはシニアの所作だ。必ずやること。）
- [ ] フレームワークの駆け足ツアー（各1日、おもちゃを作り、意見を形成する）: LangGraph、
      LlamaIndex、Vercel AI SDK、いずれかのeval/observabilityプラットフォームのSDK。それぞれ
      1段落ずつ書く: 何が得られるか、何が隠蔽されるか、どんなときに顧客に推奨するか。
- [ ] マルチプロバイダーの現実: 1つのコンポーネントをClaudeからGPTとGeminiへ移植する。すべての
      挙動の違いをField Notesに記録する — ラボのFDEは他社のモデルと競合しており、それらすべてに
      ついて流暢に語れなければならない。
